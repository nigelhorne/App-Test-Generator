   31: =head1 DESCRIPTION
   32: 
   33: Implements coverage-guided fuzzing on top of App::Test::Generator's existing
   34: schema-driven input generation.  Instead of purely random generation, it:
   35: 
   36:   1. Generates or mutates a structured input
   37:   2. Runs the target sub under Devel::Cover to capture branch hits
   38:   3. Keeps inputs that discover *new* branches in a corpus
   39:   4. Preferentially mutates corpus entries in future iterations
   40: 
   41: This is the Perl equivalent of what AFL/libFuzzer do at the byte level, but
   42: operating on typed, schema-validated Perl data structures.
   43: 
   44: =head1 HOW CORPUS FILES ARE USED
   45: 
   46: =head2 Overview
   47: 
   48: Each time C<extract-schemas --fuzz> runs, it creates or updates one JSON file
   49: per fuzzed method under C<schemas/corpus/> (or C<--corpus-dir> if specified).
   50: For example:
   51: 
   52:     schemas/corpus/translate.json
   53:     schemas/corpus/lookup.json
   54: 
   55: These files are the fuzzer's memory. Without them, every run starts from
   56: scratch. With them, each run builds on the discoveries of every previous run.
   57: 
   58: =head2 What is stored in a corpus file
   59: 
   60: Each file is a JSON object with three keys:
   61: 
   62:     {
   63:       "seed": 1234567890,
   64:       "corpus": [
   65:         { "input": "en" },
   66:         { "input": "12345678901" },
   67:         ...
   68:       ],
   69:       "bugs": [
   70:         { "input": "...", "error": "..." }
   71:       ]
   72:     }
   73: 
   74: The C<corpus> array contains every input that was judged "interesting" during
   75: past runs. An input is interesting if it triggered at least one branch in the
   76: target code that no previous input had reached. These are the inputs that
   77: proved useful for exploring the method's behaviour - not just random values,
   78: but ones that actually exercised distinct paths through the code.
   79: 
   80: The C<bugs> array records every input that caused the target method to die or
   81: throw an exception, along with the error message. This is preserved across
   82: runs so you have a permanent record of discovered failure cases even after
   83: fixing them.
   84: 
   85: The C<seed> records the random seed of the run that created the file. This is
   86: informational only and is not reused on subsequent runs.
   87: 
   88: =head2 How the corpus is used at the start of a run
   89: 
   90: When C<extract-schemas --fuzz> runs and finds an existing corpus file for a
   91: method, it calls C<load_corpus()> before starting the fuzzing loop. This
   92: pre-populates the fuzzer's internal corpus with all the previously interesting
   93: inputs. They are loaded with an empty coverage hash (C<coverage =E<gt> {}>)
   94: because coverage state from a previous process cannot be restored - only the
   95: inputs themselves are persisted.
   96: 
   97: =head2 How the corpus influences the fuzzing loop
   98: 
   99: During the main fuzzing loop, on each of the C<--fuzz-iters> iterations, the
  100: fuzzer makes a weighted random choice:
  101: 
  102: =over 4
  103: 
  104: =item 70% of iterations: mutate a corpus entry
  105: 
  106: A random entry is picked from the corpus (which now includes both the loaded
  107: entries from previous runs and any new entries discovered in this run so far).
  108: That entry's input is mutated - characters are flipped, numbers are nudged,
  109: strings are truncated or extended, array elements are duplicated or deleted -
  110: and the mutated value is run against the target method.
  111: 
  112: The key property here is that mutations are applied to inputs that are already
  113: known to reach interesting parts of the code. Rather than generating a fresh
  114: random string that will probably hit the same early conditional as everything
  115: else, the fuzzer is specifically probing the neighbourhood of inputs that
  116: previously pushed into new territory.
  117: 
  118: =item 30% of iterations: fresh random generation
  119: 
  120: A completely new input is generated from the schema. This is the exploration
  121: budget - it ensures the fuzzer does not get permanently stuck mutating a
  122: narrow slice of the input space and occasionally tries something entirely new.
  123: 
  124: =back
  125: 
  126: =head2 How new entries are added to the corpus during a run
  127: 
  128: After each input is run against the target method, the fuzzer checks whether
  129: it was interesting. With C<Devel::Cover> available, an input is interesting if
  130: it hit at least one branch that no previous input in this session had hit.
  131: Without C<Devel::Cover>, 20% of inputs are kept at random so the corpus
  132: continues to grow even without branch feedback.
  133: 
  134: Interesting inputs are appended to the in-memory corpus immediately, so they
  135: can be selected and mutated within the same run. They are also written to the
  136: JSON file at the end of the run via C<save_corpus()>.
  137: 
  138: =head2 How the corpus grows across multiple runs
  139: 
  140: On the first run, the corpus file does not exist. The fuzzer seeds itself with
  141: five randomly generated inputs, runs all iterations, and saves the interesting
  142: ones. A typical first run might produce a corpus of 15-30 entries.
  143: 
  144: On the second run, those 15-30 entries are loaded before any iteration begins.
  145: The fuzzer immediately starts mutating inputs that are already known to reach
  146: interesting branches, rather than spending iterations rediscovering them from
  147: scratch. It finds new interesting inputs on top of the existing ones, and the
  148: corpus grows further.
  149: 
  150: By the third, fourth and subsequent runs the corpus has stabilised for the
  151: easy-to-reach branches and is increasingly focused on harder-to-reach ones.
  152: The coverage plateau is reached more slowly each time, which is exactly the
  153: right behaviour - the fuzzer is spending its budget on genuinely new territory.
  154: 
  155: =head2 Practical implications
  156: 
  157: =over 4
  158: 
  159: =item Running once gives limited value; running repeatedly gives compounding value.
  160: 
  161: The first run with 100 iterations is roughly equivalent to C<App::Test::Generator>
  162: with 100 random iterations. By the fifth run, the corpus is directed at branches
  163: that purely random generation would almost never reach.
  164: 
  165: =item The corpus is human-readable and editable.
  166: 
  167: Because inputs are stored as plain JSON values, you can open a corpus file and
  168: add your own known-tricky inputs by hand. They will be picked up on the next
  169: run and mutated like any other corpus entry.
  170: 
  171: =item Deleting a corpus file resets the fuzzer for that method.
  172: 
  173: If you significantly change a method's implementation, the old corpus may be
  174: less useful. Delete the relevant C<schemas/corpus/method.json> and the fuzzer
  175: will start fresh with the new code.
  176: 
  177: =item The bugs array is a regression record.
  178: 
  179: Even after you fix a bug that was found by fuzzing, the input that triggered it
  180: remains in the C<bugs> array of the corpus file. You can use these as the basis
  181: for specific regression tests to ensure the fix holds.
  182: 
  183: =back
  184: 
  185: =head2 Corpus file location
  186: 
  187: By default corpus files are written to C<schemas/corpus/>, one file per method,
  188: named C<method_name.json>. This can be changed with the C<--corpus-dir> option:
  189: 
  190:     extract-schemas --fuzz --corpus-dir t/corpus lib/MyModule.pm
  191: 
  192: It is recommended to commit the corpus directory to version control. This means
  193: every developer and every CI run benefits from the accumulated discoveries of
  194: all previous runs rather than starting from scratch each time.
