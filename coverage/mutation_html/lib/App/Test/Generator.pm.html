  109: =head1 OVERVIEW
  110: 
  111: This module takes a formal input/output specification for a routine or
  112: method and automatically generates test cases. In effect, it allows you
  113: to easily add comprehensive black-box tests in addition to the more
  114: common white-box tests that are typically written for CPAN modules and other
  115: subroutines.
  116: 
  117: The generated tests combine:
  118: 
  119: =over 4
  120: 
  121: =item * Random fuzzing based on input types
  122: 
  123: =item * Deterministic edge cases for min/max constraints
  124: 
  125: =item * Static corpus tests defined in Perl or YAML
  126: 
  127: =back
  128: 
  129: This approach strengthens your test suite by probing both expected and
  130: unexpected inputs, helping you to catch boundary errors, invalid data
  131: handling, and regressions without manually writing every case.
  132: 
  133: =head1 DESCRIPTION
  134: 
  135: This module implements the logic behind L<fuzz-harness-generator>.
  136: It parses configuration files (fuzz and/or corpus YAML), and
  137: produces a ready-to-run F<.t> test script to run through C<prove>.
  138: 
  139: It reads configuration files in any format,
  140: and optional YAML corpus files.
  141: All of the examples in this documentation are in C<YAML> format,
  142: other formats may not work as they aren't so heavily tested.
  143: It then generates a L<Test::Most>-based fuzzing harness combining:
  144: 
  145: =over 4
  146: 
  147: =item * Randomized fuzzing of inputs (with edge cases)
  148: 
  149: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  150: 
  151: =item * Functional or OO mode (via C<$new>)
  152: 
  153: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  154: 
  155: =back
  156: 
  157: =encoding utf8
  158: 
  159: =head1 CONFIGURATION
  160: 
  161: The configuration file,
  162: for each set of tests to be produced,
  163: is a file containing a schema that can be read by L<Config::Abstraction>.

  165: =head2 SCHEMA
  166: 
  167: The schema is split into several sections.
  168: 
  169: =head3 C<%input> - input params with keys => type/optional specs
  170: 
  171: When using named parameters
  172: 
  173:   input:
  174:     name:
  175:       type: string
  176:       optional: false
  177:     age:
  178:       type: integer
  179:       optional: true
  180: 
  181: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  182: See also L<Params::Validate::Strict>.
  183: You can add more custom types using properties.
  184: 
  185: For routines with one unnamed parameter
  186: 
  187:   input:
  188:     type: string
  189: 
  190: For routines with more than one named parameter, use the C<position> keyword.
  191: 
  192:   module: Math::Simple::MinMax
  193:   fuction: max
  194: 
  195:   input:
  196:     left:
  197:       type: number
  198:       position: 0
  199:     right:
  200:       type: number
  201:       position: 1
  202: 
  203:   output:
  204:     type: number
  205: 
  206: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  207: 
  208: =head3 C<%output> - output param types for L<Return::Set> checking
  209: 
  210:   output:
  211:     type: string
  212: 
  213: If the output hash contains the key _STATUS, and if that key is set to DIES,
  214: the routine should die with the given arguments; otherwise, it should live.
  215: If it's set to WARNS,
  216: the routine should warn with the given arguments.
  217: The output can be set to the string 'undef' if the routine should return the undefined value:
  218: 
  219:   ---
  220:   module: Scalar::Util
  221:   function: blessed
  222: 
  223:   input:
  224:     type: string
  225: 
  226:   output: undef
  227: 
  228: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  229: 
  230: =head3 C<%config> - optional hash of configuration.
  231: 
  232: The current supported variables are
  233: 
  234: =over 4
  235: 
  236: =item * C<close_stdin>
  237: 
  238: Tests should not attempt to read from STDIN (default: 1).
  239: This is ignored on Windows, when never closes STDIN.
  240: 
  241: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  242: 
  243: With this test enabled, the function is expected to die when a NUL byte is passed in.
  244: 
  245: =item * C<test_undef>, test with undefined value (default: 1)
  246: 
  247: =item * C<test_empty>, test with empty strings (default: 1)
  248: 
  249: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  250: 
  251: =item * C<timeout>, ensure tests don't hang (default: 10)
  252: 
  253: Setting this to 0 disables timeout testing.
  254: 
  255: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  256: 
  257: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  258: 
  259: *item * C<test_security>, send some security string based tests (default: 0)
  260: 
  261: =back
  262: 
  263: All values default to C<true>.
  264: 
  265: =head3 C<%accessor> - this is an accessor routine
  266: 
  267:   accessor:
  268:     property: ua
  269:     type: getset
  270: 
  271: Has two mandatory elements:
  272: 
  273: =over 4
  274: 
  275: =item * C<property>
  276: 
  277: The name of the property in the object that the routine controls.
  278: 
  279: =item * C<type>
  280: 
  281: One of C<getter>, C<setter>, C<getset>.
  282: 
  283: =back
  284: 
  285: =head3 C<%transforms> - list of transformations from input sets to output sets
  286: 
  287: Transforms allow you to define how input data should be transformed into output data.
  288: This is useful for testing functions that convert between formats, normalize data,
  289: or apply business logic transformations on a set of data to different set of data.
  290: It takes a list of subsets of the input and output definitions,
  291: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  292: 
  293: =head4 Transform Validation Rules
  294: 
  295: For each transform:
  296: 
  297: =over 4
  298: 
  299: =item 1. Generate test cases using the transform's input schema
  300: 
  301: =item 2. Call the function with those inputs
  302: 
  303: =item 3. Validate the output matches the transform's output schema
  304: 
  305: =item 4. If output has a specific 'value', check exact match
  306: 
  307: =item 5. If output has constraints (min/max), validate within bounds
  308: 
  309: =back

  578: =head2 EDGE CASE GENERATION
  579: 
  580: In addition to purely random fuzz cases, the harness generates
  581: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  582: 
  583: For each constraint, three edge cases are added:
  584: 
  585: =over 4
  586: 
  587: =item * Just inside the allowable range
  588: 
  589: This case should succeed, since it lies strictly within the bounds.
  590: 
  591: =item * Exactly on the boundary
  592: 
  593: This case should succeed, since it meets the constraint exactly.
  594: 
  595: =item * Just outside the boundary
  596: 
  597: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  598: should cause the harness to fail validation or croak.
  599: 
  600: =back
  601: 
  602: Supported constraint types:
  603: 
  604: =over 4
  605: 
  606: =item * C<number>, C<integer>, C<float>
  607: 
  608: Uses numeric values one below, equal to, and one above the boundary.
  609: 
  610: =item * C<string>
  611: 
  612: Uses strings of lengths one below, equal to, and one above the boundary.
  613: 
  614: =item * C<arrayref>
  615: 
  616: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  617: 
  618: =item * C<hashref>
  619: 
  620: Uses hashes with key counts one below, equal to, and one above the
  621: boundary (C<min> = minimum number of keys, C<max> = maximum number
  622: of keys).
  623: 
  624: =item * C<memberof> - arrayref of allowed values for a parameter
  625: 
  626: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  627: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  628: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  629: 
  630:   ---
  631:   input:
  632:     status:
  633:       type: string
  634:       memberof:
  635:         - ok
  636:         - error
  637:         - pending
  638:     level:
  639:       type: integer
  640:       memberof:
  641:         - 1
  642:         - 5
  643:         - 111
  644: 
  645: The generator will automatically create test cases for each allowed value (inside the member list),
  646: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  647: This works for strings, integers, and numbers.
  648: 
  649: =item * C<enum> - synonym of C<memberof>
  650: 
  651: =item * C<boolean> - automatic boundary tests for boolean fields
  652: 
  653:   input:
  654:     flag:
  655:       type: boolean
  656: 
  657: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  658: 
  659: =back
  660: 
  661: These edge cases are inserted automatically, in addition to the random
  662: fuzzing inputs, so each run will reliably probe boundary conditions
  663: without relying solely on randomness.

  823: =head2 Property-Based Testing with Transforms
  824: 
  825: The generator can create property-based tests using L<Test::LectroTest> when the
  826: C<properties> configuration option is enabled.
  827: This provides more comprehensive
  828: testing by automatically generating thousands of test cases and verifying that
  829: mathematical properties hold across all inputs.
  830: 
  831: =head3 Basic Property-Based Transform Example
  832: 
  833: Here's a complete example testing the C<abs> builtin function:
  834: 
  835: B<t/conf/abs.yml>:
  836: 
  837:   ---
  838:   module: builtin
  839:   function: abs
  840: 
  841:   config:
  842:     test_undef: no
  843:     test_empty: no
  844:     test_nuls: no
  845:     properties:
  846:       enable: true
  847:       trials: 1000
  848: 
  849:   input:
  850:     number:
  851:       type: number
  852:       position: 0
  853: 
  854:   output:
  855:     type: number
  856:     min: 0
  857: 
  858:   transforms:
  859:     positive:
  860:       input:
  861:         number:
  862:           type: number
  863:           min: 0
  864:       output:
  865:         type: number
  866:         min: 0
  867: 
  868:     negative:
  869:       input:
  870:         number:
  871:           type: number
  872:           max: 0
  873:       output:
  874:         type: number
  875:         min: 0

  878: 
  879: =over 4
  880: 
  881: =item * Enables property-based testing with 1000 trials per property
  882: 
  883: =item * Defines two transforms: one for positive numbers, one for negative
  884: 
  885: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
  886:

  890: 
  891:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t

  894:

  896: 
  897: =item * Traditional edge-case tests for boundary conditions

  901: =item * Property-based tests that verify the transforms with 1000 trials each
  902: 
  903: =back

  905: =head3 What Properties Are Tested?
  906: 
  907: The generator automatically detects and tests these properties based on your transform specifications:
  908: 
  909: =over 4
  910: 
  911: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
  912: 
  913: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
  914: 
  915: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
  916: 
  917: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
  918: 
  919: =back

  922: 
  923:   # For the "positive" transform:
  924:   - Given a positive number, abs() returns >= 0

  926:   - The result is defined
  927: 
  928:   # For the "negative" transform:
  929:   - Given a negative number, abs() returns >= 0
  930:   - The result is a valid number
  931:   - The result is defined
