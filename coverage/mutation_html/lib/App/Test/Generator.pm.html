   96: =head1 OVERVIEW
   97: 
   98: This module takes a formal input/output specification for a routine or
   99: method and automatically generates test cases. In effect, it allows you
  100: to easily add comprehensive black-box tests in addition to the more
  101: common white-box tests that are typically written for CPAN modules and other
  102: subroutines.
  103: 
  104: The generated tests combine:
  105: 
  106: =over 4
  107: 
  108: =item * Random fuzzing based on input types
  109: 
  110: =item * Deterministic edge cases for min/max constraints
  111: 
  112: =item * Static corpus tests defined in Perl or YAML
  113: 
  114: =back
  115: 
  116: This approach strengthens your test suite by probing both expected and
  117: unexpected inputs, helping you to catch boundary errors, invalid data
  118: handling, and regressions without manually writing every case.
  119: 
  120: =head1 DESCRIPTION
  121: 
  122: This module implements the logic behind L<fuzz-harness-generator>.
  123: It parses configuration files (fuzz and/or corpus YAML), and
  124: produces a ready-to-run F<.t> test script to run through C<prove>.
  125: 
  126: It reads configuration files in any format,
  127: and optional YAML corpus files.
  128: All of the examples in this documentation are in C<YAML> format,
  129: other formats may not work as they aren't so heavily tested.
  130: It then generates a L<Test::Most>-based fuzzing harness combining:
  131: 
  132: =over 4
  133: 
  134: =item * Randomized fuzzing of inputs (with edge cases)
  135: 
  136: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  137: 
  138: =item * Functional or OO mode (via C<$new>)
  139: 
  140: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  141: 
  142: =back
  143: 
  144: =head1 CONFIGURATION
  145: 
  146: The configuration file,
  147: for each set of tests to be produced,
  148: is a file containing a schema that can be read by L<Config::Abstraction>.

  150: =head2 SCHEMA
  151: 
  152: The schema is split into several sections.
  153: 
  154: =head3 C<%input> - input params with keys => type/optional specs
  155: 
  156: When using named parameters
  157: 
  158:   input:
  159:     name:
  160:       type: string
  161:       optional: false
  162:     age:
  163:       type: integer
  164:       optional: true
  165: 
  166: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  167: See also L<Params::Validate::Strict>.
  168: You can add more custom types using properties.
  169: 
  170: For routines with one unnamed parameter
  171: 
  172:   input:
  173:     type: string
  174: 
  175: For routines with more than one named parameter, use the C<position> keyword.
  176: 
  177:   module: Math::Simple::MinMax
  178:   fuction: max
  179: 
  180:   input:
  181:     left:
  182:       type: number
  183:       position: 0
  184:     right:
  185:       type: number
  186:       position: 1
  187: 
  188:   output:
  189:     type: number
  190: 
  191: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  192: 
  193: =head3 C<%output> - output param types for L<Return::Set> checking
  194: 
  195:   output:
  196:     type: string
  197: 
  198: If the output hash contains the key _STATUS, and if that key is set to DIES,
  199: the routine should die with the given arguments; otherwise, it should live.
  200: If it's set to WARNS,
  201: the routine should warn with the given arguments.
  202: The output can be set to the string 'undef' if the routine should return the undefined value:
  203: 
  204:   ---
  205:   module: Scalar::Util
  206:   function: blessed
  207: 
  208:   input:
  209:     type: string
  210: 
  211:   output: undef
  212: 
  213: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  214: 
  215: =head3 C<%config> - optional hash of configuration.
  216: 
  217: The current supported variables are
  218: 
  219: =over 4
  220: 
  221: =item * C<close_stdin>
  222: 
  223: Tests should not attempt to read from STDIN (default: 1).
  224: This is ignored on Windows, when never closes STDIN.
  225: 
  226: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  227: 
  228: With this test enabled, the function is expected to die when a NUL byte is passed in.
  229: 
  230: =item * C<test_undef>, test with undefined value (default: 1)
  231: 
  232: =item * C<test_empty>, test with empty strings (default: 1)
  233: 
  234: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  235: 
  236: =item * C<timeout>, ensure tests don't hang (default: 10)
  237: 
  238: Setting this to 0 disables timeout testing.
  239: 
  240: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  241: 
  242: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  243: 
  244: *item * C<test_security>, send some security string based tests (default: 0)
  245: 
  246: =back
  247: 
  248: All values default to C<true>.
  249: 
  250: =head3 C<%accessor> - this is an accessor routine
  251: 
  252:   accessor:
  253:     property: ua
  254:     type: getset
  255: 
  256: Has two mandatory elements:
  257: 
  258: =over 4
  259: 
  260: =item * C<property>
  261: 
  262: The name of the property in the object that the routine controls.
  263: 
  264: =item * C<type>
  265: 
  266: One of C<getter>, C<setter>, C<getset>.
  267: 
  268: =back
  269: 
  270: =head3 C<%transforms> - list of transformations from input sets to output sets
  271: 
  272: Transforms allow you to define how input data should be transformed into output data.
  273: This is useful for testing functions that convert between formats, normalize data,
  274: or apply business logic transformations on a set of data to different set of data.
  275: It takes a list of subsets of the input and output definitions,
  276: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  277: 
  278: =head4 Transform Validation Rules
  279: 
  280: For each transform:
  281: 
  282: =over 4
  283: 
  284: =item 1. Generate test cases using the transform's input schema
  285: 
  286: =item 2. Call the function with those inputs
  287: 
  288: =item 3. Validate the output matches the transform's output schema
  289: 
  290: =item 4. If output has a specific 'value', check exact match
  291: 
  292: =item 5. If output has constraints (min/max), validate within bounds
  293: 
  294: =back

  563: =head2 EDGE CASE GENERATION
  564: 
  565: In addition to purely random fuzz cases, the harness generates
  566: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  567: 
  568: For each constraint, three edge cases are added:
  569: 
  570: =over 4
  571: 
  572: =item * Just inside the allowable range
  573: 
  574: This case should succeed, since it lies strictly within the bounds.
  575: 
  576: =item * Exactly on the boundary
  577: 
  578: This case should succeed, since it meets the constraint exactly.
  579: 
  580: =item * Just outside the boundary
  581: 
  582: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  583: should cause the harness to fail validation or croak.
  584: 
  585: =back
  586: 
  587: Supported constraint types:
  588: 
  589: =over 4
  590: 
  591: =item * C<number>, C<integer>, C<float>
  592: 
  593: Uses numeric values one below, equal to, and one above the boundary.
  594: 
  595: =item * C<string>
  596: 
  597: Uses strings of lengths one below, equal to, and one above the boundary.
  598: 
  599: =item * C<arrayref>
  600: 
  601: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  602: 
  603: =item * C<hashref>
  604: 
  605: Uses hashes with key counts one below, equal to, and one above the
  606: boundary (C<min> = minimum number of keys, C<max> = maximum number
  607: of keys).
  608: 
  609: =item * C<memberof> - arrayref of allowed values for a parameter
  610: 
  611: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  612: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  613: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  614: 
  615:   ---
  616:   input:
  617:     status:
  618:       type: string
  619:       memberof:
  620:         - ok
  621:         - error
  622:         - pending
  623:     level:
  624:       type: integer
  625:       memberof:
  626:         - 1
  627:         - 5
  628:         - 111
  629: 
  630: The generator will automatically create test cases for each allowed value (inside the member list),
  631: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  632: This works for strings, integers, and numbers.
  633: 
  634: =item * C<enum> - synonym of C<memberof>
  635: 
  636: =item * C<boolean> - automatic boundary tests for boolean fields
  637: 
  638:   input:
  639:     flag:
  640:       type: boolean
  641: 
  642: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  643: 
  644: =back
  645: 
  646: These edge cases are inserted automatically, in addition to the random
  647: fuzzing inputs, so each run will reliably probe boundary conditions
  648: without relying solely on randomness.

  808: =head2 Property-Based Testing with Transforms
  809: 
  810: The generator can create property-based tests using L<Test::LectroTest> when the
  811: C<properties> configuration option is enabled.
  812: This provides more comprehensive
  813: testing by automatically generating thousands of test cases and verifying that
  814: mathematical properties hold across all inputs.
  815: 
  816: =head3 Basic Property-Based Transform Example
  817: 
  818: Here's a complete example testing the C<abs> builtin function:
  819: 
  820: B<t/conf/abs.yml>:
  821: 
  822:   ---
  823:   module: builtin
  824:   function: abs
  825: 
  826:   config:
  827:     test_undef: no
  828:     test_empty: no
  829:     test_nuls: no
  830:     properties:
  831:       enable: true
  832:       trials: 1000
  833: 
  834:   input:
  835:     number:
  836:       type: number
  837:       position: 0
  838: 
  839:   output:
  840:     type: number
  841:     min: 0
  842: 
  843:   transforms:
  844:     positive:
  845:       input:
  846:         number:
  847:           type: number
  848:           min: 0
  849:       output:
  850:         type: number
  851:         min: 0
  852: 
  853:     negative:
  854:       input:
  855:         number:
  856:           type: number
  857:           max: 0
  858:       output:
  859:         type: number
  860:         min: 0
  861:

  863: 
  864: =over 4
  865: 
  866: =item * Enables property-based testing with 1000 trials per property
  867: 
  868: =item * Defines two transforms: one for positive numbers, one for negative
  869: 
  870: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
  871: 
  872: =back

  874: Generate the test:
  875: 
  876:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t

  878: The generated test will include:
  879: 
  880: =over 4
  881: 
  882: =item * Traditional edge-case tests for boundary conditions
  883: 
  884: =item * Random fuzzing with 30 iterations (or as configured)
  885: 
  886: =item * Property-based tests that verify the transforms with 1000 trials each
  887: 
  888: =back

  890: =head3 What Properties Are Tested?
  891: 
  892: The generator automatically detects and tests these properties based on your transform specifications:
  893: 
  894: =over 4
  895: 
  896: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
  897: 
  898: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
  899: 
  900: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
  901: 
  902: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
  903: 
  904: =back
  905: 
  906: For the C<abs> example above, the generated properties verify:
  907: 
  908:   # For the "positive" transform:
  909:   - Given a positive number, abs() returns >= 0
  910:   - The result is a valid number
  911:   - The result is defined
  912: 
  913:   # For the "negative" transform:
  914:   - Given a negative number, abs() returns >= 0
  915:   - The result is a valid number
  916:   - The result is defined
