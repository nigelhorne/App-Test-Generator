  228: =head1 OVERVIEW
  229: 
  230: This module takes a formal input/output specification for a routine or
  231: method and automatically generates test cases. In effect, it allows you
  232: to easily add comprehensive black-box tests in addition to the more
  233: common white-box tests that are typically written for CPAN modules and other
  234: subroutines.
  235: 
  236: The generated tests combine:
  237: 
  238: =over 4
  239: 
  240: =item * Random fuzzing based on input types
  241: 
  242: =item * Deterministic edge cases for min/max constraints
  243: 
  244: =item * Static corpus tests defined in Perl or YAML
  245: 
  246: =back
  247: 
  248: This approach strengthens your test suite by probing both expected and
  249: unexpected inputs, helping you to catch boundary errors, invalid data
  250: handling, and regressions without manually writing every case.
  251: 
  252: =head1 TOOLS
  253: 
  254: The distribution ships the following command-line tools:
  255: 
  256: =over 4
  257: 
  258: =item * L<benchmark-generator> - generate a self-contained L<Benchmark> C<cmpthese> script from a YAML schema. Each transform in the schema becomes one named variant; representative input values are derived from each parameter's type and range constraints.
  259: 
  260: =item * L<deploy-workflows> - copy C<dashboard.yml> and C<mutate.yml> into the target repository's C<.github/workflows/> directory. Both files are embedded verbatim in the script, so no ATG source tree is needed after installation. Supports C<--target>, C<--force>, and C<--dry-run>.
  261: 
  262: =item * L<extract-schemas> - heuristically extract YAML parameter schemas from a C<.pm> file, with optional coverage-guided fuzzing (C<--fuzz>) and corpus minimization (C<--minimize-corpus>).
  263: 
  264: =item * L<fuzz-harness-generator> - generate a C<Test::Most> fuzzing harness from a YAML schema.
  265: 
  266: =item * L<pod-example-tester> - generate a C<Test::Most> round-trip test file from a module's POD code examples. Annotated examples (C<# returns value> / C<< # => value >>) get C<is()> assertions; unannotated verbatim blocks are wrapped in C<eval{}> and checked for no exception.
  267: 
  268: =item * L<test-generator-mutate> - run mutation testing against a module's test suite.
  269: 
  270: =item * L<test-generator-index> - generate the HTML test-quality dashboard, combining Devel::Cover statement/branch data, LCSAJ path coverage, mutation results, and CPAN Testers failure analysis. For each CPAN Testers FAIL report, also writes a self-contained shell script (C<cover_html/reproduce/reproduce-GUID.sh>) that pins every installed module at its exact failing version, enabling local reproduction of the failure environment.
  271: 
  272: =back

  274: =head1 DESCRIPTION
  275: 
  276: This module implements the logic behind L<fuzz-harness-generator>.
  277: It parses configuration files (fuzz and/or corpus YAML), and
  278: produces a ready-to-run F<.t> test script to run through C<prove>.
  279: 
  280: It reads configuration files in any format,
  281: and optional YAML corpus files.
  282: All of the examples in this documentation are in C<YAML> format,
  283: other formats may not work as they aren't so heavily tested.
  284: It then generates a L<Test::Most>-based fuzzing harness combining:
  285: 
  286: =over 4
  287: 
  288: =item * Randomized fuzzing of inputs (with edge cases)
  289: 
  290: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  291: 
  292: =item * Functional or OO mode (via C<$new>)
  293: 
  294: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  295: 
  296: =back

  298: =head1 MUTATION-GUIDED TEST GENERATION
  299: 
  300: C<App::Test::Generator> includes a pipeline that automatically closes the
  301: feedback loop between mutation testing, schema extraction, and fuzz
  302: testing. The goal is that surviving mutants drive the creation of new
  303: tests that kill them on the next run, without manual intervention.
  304: 
  305: =head2 The Pipeline
  306: 
  307:     mutation survivor
  308:         |
  309:         v
  310:     SchemaExtractor extracts the schema for the enclosing sub
  311:         |
  312:         v
  313:     Schema augmented with boundary values from the mutant
  314:         |
  315:         v
  316:     Augmented schema written to t/conf/
  317:         |
  318:         v
  319:     t/fuzz.t picks up the new schema and runs fuzz tests
  320:         |
  321:         v
  322:     Mutation killed on next run
  323: 
  324: =head2 How to Use It
  325: 
  326: The pipeline is driven by three flags passed to
  327: C<bin/test-generator-index>, which is invoked automatically by
  328: C<bin/generate-test-dashboard> on each CI push.
  329: 
  330: =head3 Step 1: Generate TODO stubs for all survivors
  331: 
  332:     bin/test-generator-index --generate_mutant_tests=t
  333: 
  334: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  335: 
  336: =over 4
  337: 
  338: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  339: boundary value suggestions, environment variable hints, and the
  340: enclosing subroutine name for navigation context.
  341: 
  342: =item * Comment-only hints for LOW difficulty survivors.
  343: 
  344: =back
  345: 
  346: Multiple mutations on the same source line are deduplicated into one
  347: stub. One good test kills all variants on that line.
  348: 
  349: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  350: 
  351:     bin/test-generator-index \
  352:         --generate_mutant_tests=t \
  353:         --generate_test=mutant
  354: 
  355: For each NUM_BOUNDARY survivor, calls
  356: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  357: the enclosing subroutine. If the confidence level is sufficient, the
  358: schema is augmented with the boundary value from the mutant (plus one
  359: value either side) and written to C<t/conf/> as a runnable YAML file.
  360: L<t/fuzz.t> picks it up automatically on the next test run.
  361: 
  362: Falls back to a TODO stub if:
  363: 
  364: =over 4
  365: 
  366: =item * SchemaExtractor cannot parse the file
  367: 
  368: =item * The enclosing sub cannot be determined
  369: 
  370: =item * The extracted schema confidence is C<very_low> or C<none>
  371: 
  372: =back
  373: 
  374: =head3 Step 3: Augment existing schemas with survivor boundary values
  375: 
  376:     bin/test-generator-index \
  377:         --generate_mutant_tests=t \
  378:         --generate_test=mutant \
  379:         --generate_fuzz
  380: 
  381: Scans C<t/conf/> for existing YAML schema files (hand-written or
  382: previously generated) and writes augmented copies with boundary values
  383: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  384: never modified. Augmented copies are written as
  385: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  386: automatically by C<t/fuzz.t>.
  387: 
  388: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  389: to prevent cascading augmentation. Schemas with no matching survivors
  390: are skipped, with a note if C<--verbose> is active.
  391: 
  392: =head3 Putting It All Together
  393: 
  394: The recommended invocation in C<bin/generate-test-dashboard>
  395: Step 7 runs all three stages together:
  396: 
  397:     bin/test-generator-index \
  398:         --generate_mutant_tests=t \
  399:         --generate_test=mutant \
  400:         --generate_fuzz
  401: 
  402: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  403: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  404: to the repository so they accumulate over time as the test suite
  405: improves.

  407: =head2 Confidence Levels
  408: 
  409: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  410: to each extracted schema:
  411: 
  412: =over 4
  413: 
  414: =item * C<high> / C<medium> / C<low> - Schema is used for test generation

  422: with explicit parameter validation (L<Params::Validate::Strict>,
  423: L<Params::Get>) or comprehensive POD will produce higher-confidence

  464: 
  465: The configuration file,
  466: for each set of tests to be produced,
  467: is a file containing a schema that can be read by L<Config::Abstraction>.

  469: =head2 SCHEMA
  470: 
  471: The schema is split into several sections.
  472: 
  473: =head3 C<%input> - input params with keys => type/optional specs
  474: 
  475: When using named parameters
  476: 
  477:   input:
  478:     name:
  479:       type: string
  480:       optional: false
  481:     age:
  482:       type: integer
  483:       optional: true
  484: 
  485: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  486: See also L<Params::Validate::Strict>.
  487: You can add more custom types using properties.
  488: 
  489: For routines with one unnamed parameter
  490: 
  491:   input:
  492:     type: string
  493: 
  494: For routines with more than one named parameter, use the C<position> keyword.
  495: 
  496:   module: Math::Simple::MinMax
  497:   fuction: max
  498: 
  499:   input:
  500:     left:
  501:       type: number
  502:       position: 0
  503:     right:
  504:       type: number
  505:       position: 1
  506: 
  507:   output:
  508:     type: number
  509: 
  510: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  511: 
  512: =head3 C<%output> - output param types for L<Return::Set> checking
  513: 
  514:   output:
  515:     type: string
  516: 
  517: If the output hash contains the key _STATUS, and if that key is set to DIES,
  518: the routine should die with the given arguments; otherwise, it should live.
  519: If it's set to WARNS,
  520: the routine should warn with the given arguments.
  521: The output can be set to the string 'undef' if the routine should return the undefined value:
  522: 
  523:   ---
  524:   module: Scalar::Util
  525:   function: blessed
  526: 
  527:   input:
  528:     type: string
  529: 
  530:   output: undef
  531: 
  532: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  533: 
  534: For methods that return a list (rather than a reference), use C<type: array>.
  535: The generated test captures the result in list context and validates it as an
  536: arrayref, which requires L<Test::Returns> 0.03 or later:
  537: 
  538:   output:
  539:     type: array
  540: 
  541: =head3 C<%config> - optional hash of configuration.
  542: 
  543: The current supported variables are
  544: 
  545: =over 4
  546: 
  547: =item * C<close_stdin>
  548: 
  549: Tests should not attempt to read from STDIN (default: 1).
  550: This is ignored on Windows, when never closes STDIN.
  551: 
  552: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  553: 
  554: With this test enabled, the function is expected to die when a NUL byte is passed in.
  555: 
  556: =item * C<test_undef>, test with undefined value (default: 1)
  557: 
  558: =item * C<test_empty>, test with empty strings (default: 1)
  559: 
  560: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  561: 
  562: =item * C<timeout>, ensure tests don't hang (default: 10)
  563: 
  564: Setting this to 0 disables timeout testing.
  565: 
  566: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  567: 
  568: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  569: 
  570: *item * C<test_security>, send some security string based tests (default: 0)
  571: 
  572: =back
  573: 
  574: All values default to C<true>.
  575: 
  576: =head3 C<%accessor> - this is an accessor routine
  577: 
  578:   accessor:
  579:     property: ua
  580:     type: getset
  581: 
  582: Has two mandatory elements:
  583: 
  584: =over 4
  585: 
  586: =item * C<property>
  587: 
  588: The name of the property in the object that the routine controls.
  589: 
  590: =item * C<type>
  591: 
  592: One of C<getter>, C<setter>, C<getset>.
  593: 
  594: =back
  595: 
  596: =head3 C<%transforms> - list of transformations from input sets to output sets
  597: 
  598: Transforms allow you to define how input data should be transformed into output data.
  599: This is useful for testing functions that convert between formats, normalize data,
  600: or apply business logic transformations on a set of data to different set of data.
  601: It takes a list of subsets of the input and output definitions,
  602: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  603: 
  604: =head4 Transform Validation Rules
  605: 
  606: For each transform:
  607: 
  608: =over 4
  609: 
  610: =item 1. Generate test cases using the transform's input schema
  611: 
  612: =item 2. Call the function with those inputs
  613: 
  614: =item 3. Validate the output matches the transform's output schema
  615: 
  616: =item 4. If output has a specific 'value', check exact match

  889: =head2 EDGE CASE GENERATION
  890: 
  891: In addition to purely random fuzz cases, the harness generates
  892: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  893: 
  894: For each constraint, three edge cases are added:
  895: 
  896: =over 4
  897: 
  898: =item * Just inside the allowable range
  899: 
  900: This case should succeed, since it lies strictly within the bounds.
  901: 
  902: =item * Exactly on the boundary
  903: 
  904: This case should succeed, since it meets the constraint exactly.
  905: 
  906: =item * Just outside the boundary
  907: 
  908: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  909: should cause the harness to fail validation or croak.
  910: 
  911: =back
  912: 
  913: Supported constraint types:
  914: 
  915: =over 4
  916: 
  917: =item * C<number>, C<integer>, C<float>
  918: 
  919: Uses numeric values one below, equal to, and one above the boundary.
  920: 
  921: =item * C<string>
  922: 
  923: Uses strings of lengths one below, equal to, and one above the boundary.
  924: 
  925: =item * C<arrayref>
  926: 
  927: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  928: 
  929: =item * C<hashref>
  930: 
  931: Uses hashes with key counts one below, equal to, and one above the
  932: boundary (C<min> = minimum number of keys, C<max> = maximum number
  933: of keys).
  934: 
  935: =item * C<memberof> - arrayref of allowed values for a parameter
  936: 
  937: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  938: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  939: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  940: 
  941:   ---
  942:   input:
  943:     status:
  944:       type: string
  945:       memberof:
  946:         - ok
  947:         - error
  948:         - pending
  949:     level:
  950:       type: integer
  951:       memberof:
  952:         - 1
  953:         - 5
  954:         - 111
  955: 
  956: The generator will automatically create test cases for each allowed value (inside the member list),
  957: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  958: This works for strings, integers, and numbers.
  959: 
  960: =item * C<enum> - synonym of C<memberof>
  961: 
  962: =item * C<boolean> - automatic boundary tests for boolean fields
  963: 
  964:   input:
  965:     flag:
  966:       type: boolean
  967: 
  968: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  969: 
  970: =back
  971: 
  972: These edge cases are inserted automatically, in addition to the random
  973: fuzzing inputs, so each run will reliably probe boundary conditions
  974: without relying solely on randomness.

 1134: =head2 Property-Based Testing with Transforms
 1135: 
 1136: The generator can create property-based tests using L<Test::LectroTest> when the
 1137: C<properties> configuration option is enabled.
 1138: This provides more comprehensive
 1139: testing by automatically generating thousands of test cases and verifying that
 1140: mathematical properties hold across all inputs.
 1141: 
 1142: =head3 Basic Property-Based Transform Example
 1143: 
 1144: Here's a complete example testing the C<abs> builtin function:
 1145: 
 1146: B<t/conf/abs.yml>:
 1147: 
 1148:   ---
 1149:   module: builtin
 1150:   function: abs
 1151: 
 1152:   config:
 1153:     test_undef: no
 1154:     test_empty: no
 1155:     test_nuls: no
 1156:     properties:
 1157:       enable: true
 1158:       trials: 1000
 1159: 
 1160:   input:
 1161:     number:
 1162:       type: number
 1163:       position: 0
 1164: 
 1165:   output:
 1166:     type: number
 1167:     min: 0
 1168: 
 1169:   transforms:
 1170:     positive:
 1171:       input:
 1172:         number:
 1173:           type: number
 1174:           min: 0
 1175:       output:
 1176:         type: number
 1177:         min: 0
 1178: 
 1179:     negative:
 1180:       input:
 1181:         number:
 1182:           type: number
 1183:           max: 0
 1184:       output:
 1185:         type: number
 1186:         min: 0
 1187: 
 1188: This configuration:
 1189: 
 1190: =over 4
 1191: 
 1192: =item * Enables property-based testing with 1000 trials per property
 1193: 
 1194: =item * Defines two transforms: one for positive numbers, one for negative
 1195: 
 1196: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1197: 
 1198: =back
 1199: 
 1200: Generate the test:
 1201: 
 1202:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t
 1203: 
 1204: The generated test will include:
 1205: 
 1206: =over 4
 1207: 
 1208: =item * Traditional edge-case tests for boundary conditions
 1209: 
 1210: =item * Random fuzzing with 30 iterations (or as configured)
 1211: 
 1212: =item * Property-based tests that verify the transforms with 1000 trials each
 1213: 
 1214: =back
 1215: 
 1216: =head3 What Properties Are Tested?
 1217: 
 1218: The generator automatically detects and tests these properties based on your transform specifications:
 1219: 
 1220: =over 4
 1221: 
 1222: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1223: 
 1224: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1225: 
 1226: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1227: 
 1228: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
 1229: 
 1230: =back
 1231: 
 1232: For the C<abs> example above, the generated properties verify:
 1233: 
 1234:   # For the "positive" transform:
 1235:   - Given a positive number, abs() returns >= 0
 1236:   - The result is a valid number
 1237:   - The result is defined
 1238: 
 1239:   # For the "negative" transform:
 1240:   - Given a negative number, abs() returns >= 0
 1241:   - The result is a valid number
 1242:   - The result is defined
