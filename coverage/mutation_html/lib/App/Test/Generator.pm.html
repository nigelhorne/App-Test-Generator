  109: =head1 OVERVIEW
  110: 
  111: This module takes a formal input/output specification for a routine or
  112: method and automatically generates test cases. In effect, it allows you
  113: to easily add comprehensive black-box tests in addition to the more
  114: common white-box tests that are typically written for CPAN modules and other
  115: subroutines.
  116: 
  117: The generated tests combine:
  118: 
  119: =over 4
  120: 
  121: =item * Random fuzzing based on input types
  122: 
  123: =item * Deterministic edge cases for min/max constraints
  124: 
  125: =item * Static corpus tests defined in Perl or YAML
  126: 
  127: =back
  128: 
  129: This approach strengthens your test suite by probing both expected and
  130: unexpected inputs, helping you to catch boundary errors, invalid data
  131: handling, and regressions without manually writing every case.
  132: 
  133: =head1 DESCRIPTION
  134: 
  135: This module implements the logic behind L<fuzz-harness-generator>.
  136: It parses configuration files (fuzz and/or corpus YAML), and
  137: produces a ready-to-run F<.t> test script to run through C<prove>.
  138: 
  139: It reads configuration files in any format,
  140: and optional YAML corpus files.
  141: All of the examples in this documentation are in C<YAML> format,
  142: other formats may not work as they aren't so heavily tested.
  143: It then generates a L<Test::Most>-based fuzzing harness combining:
  144: 
  145: =over 4
  146: 
  147: =item * Randomized fuzzing of inputs (with edge cases)
  148: 
  149: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  150: 
  151: =item * Functional or OO mode (via C<$new>)
  152: 
  153: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  154: 
  155: =back
  156: 
  157: =head1 MUTATION-GUIDED TEST GENERATION
  158: 
  159: C<App::Test::Generator> includes a pipeline that automatically closes the
  160: feedback loop between mutation testing, schema extraction, and fuzz
  161: testing. The goal is that surviving mutants drive the creation of new
  162: tests that kill them on the next run, without manual intervention.
  163: 
  164: =head2 The Pipeline
  165: 
  166:     mutation survivor
  167:         |
  168:         v
  169:     SchemaExtractor extracts the schema for the enclosing sub
  170:         |
  171:         v
  172:     Schema augmented with boundary values from the mutant
  173:         |
  174:         v
  175:     Augmented schema written to t/conf/
  176:         |
  177:         v
  178:     t/fuzz.t picks up the new schema and runs fuzz tests
  179:         |
  180:         v
  181:     Mutation killed on next run
  182: 
  183: =head2 How to Use It
  184: 
  185: The pipeline is driven by three flags passed to
  186: C<scripts/generate_index.pl>, which is invoked automatically by
  187: C<scripts/generate_test_dashboard> on each CI push.
  188: 
  189: =head3 Step 1: Generate TODO stubs for all survivors
  190: 
  191:     scripts/generate_index.pl --generate_mutant_tests=t
  192: 
  193: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  194: 
  195: =over 4
  196: 
  197: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  198: boundary value suggestions, environment variable hints, and the
  199: enclosing subroutine name for navigation context.
  200: 
  201: =item * Comment-only hints for LOW difficulty survivors.
  202: 
  203: =back
  204: 
  205: Multiple mutations on the same source line are deduplicated into one
  206: stub. One good test kills all variants on that line.
  207: 
  208: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  209: 
  210:     scripts/generate_index.pl \
  211:         --generate_mutant_tests=t \
  212:         --generate_test=mutant
  213: 
  214: For each NUM_BOUNDARY survivor, calls
  215: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  216: the enclosing subroutine. If the confidence level is sufficient, the
  217: schema is augmented with the boundary value from the mutant (plus one
  218: value either side) and written to C<t/conf/> as a runnable YAML file.
  219: L<t/fuzz.t> picks it up automatically on the next test run.
  220: 
  221: Falls back to a TODO stub if:
  222: 
  223: =over 4
  224: 
  225: =item * SchemaExtractor cannot parse the file
  226: 
  227: =item * The enclosing sub cannot be determined
  228: 
  229: =item * The extracted schema confidence is C<very_low> or C<none>
  230: 
  231: =back
  232: 
  233: =head3 Step 3: Augment existing schemas with survivor boundary values
  234: 
  235:     scripts/generate_index.pl \
  236:         --generate_mutant_tests=t \
  237:         --generate_test=mutant \
  238:         --generate_fuzz
  239: 
  240: Scans C<t/conf/> for existing YAML schema files (hand-written or
  241: previously generated) and writes augmented copies with boundary values
  242: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  243: never modified. Augmented copies are written as
  244: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  245: automatically by C<t/fuzz.t>.
  246: 
  247: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  248: to prevent cascading augmentation. Schemas with no matching survivors
  249: are skipped, with a note if C<--verbose> is active.
  250: 
  251: =head3 Putting It All Together
  252: 
  253: The recommended invocation in C<scripts/generate_test_dashboard>
  254: Step 7 runs all three stages together:
  255: 
  256:     scripts/generate_index.pl \
  257:         --generate_mutant_tests=t \
  258:         --generate_test=mutant \
  259:         --generate_fuzz
  260: 
  261: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  262: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  263: to the repository so they accumulate over time as the test suite
  264: improves.

  266: =head2 Confidence Levels
  267: 
  268: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  269: to each extracted schema:
  270: 
  271: =over 4
  272: 
  273: =item * C<high> / C<medium> / C<low> - Schema is used for test generation
  274: 
  275: =item * C<very_low> / C<none> - Falls back to TODO stub
  276: 
  277: =back
  278: 
  279: Confidence is based on how much type and constraint information could
  280: be inferred from the source code and its POD documentation. Methods
  281: with explicit parameter validation (L<Params::Validate::Strict>,
  282: L<Params::Get>) or comprehensive POD will produce higher-confidence
  283: schemas.

  322: =head1 CONFIGURATION
  323: 
  324: The configuration file,
  325: for each set of tests to be produced,
  326: is a file containing a schema that can be read by L<Config::Abstraction>.
  327: 
  328: =head2 SCHEMA
  329: 
  330: The schema is split into several sections.
  331: 
  332: =head3 C<%input> - input params with keys => type/optional specs
  333: 
  334: When using named parameters
  335: 
  336:   input:
  337:     name:
  338:       type: string
  339:       optional: false
  340:     age:
  341:       type: integer
  342:       optional: true
  343: 
  344: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  345: See also L<Params::Validate::Strict>.
  346: You can add more custom types using properties.
  347: 
  348: For routines with one unnamed parameter
  349: 
  350:   input:
  351:     type: string
  352: 
  353: For routines with more than one named parameter, use the C<position> keyword.
  354: 
  355:   module: Math::Simple::MinMax
  356:   fuction: max
  357: 
  358:   input:
  359:     left:
  360:       type: number
  361:       position: 0
  362:     right:
  363:       type: number
  364:       position: 1
  365: 
  366:   output:
  367:     type: number
  368: 
  369: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  370: 
  371: =head3 C<%output> - output param types for L<Return::Set> checking
  372: 
  373:   output:
  374:     type: string
  375: 
  376: If the output hash contains the key _STATUS, and if that key is set to DIES,
  377: the routine should die with the given arguments; otherwise, it should live.
  378: If it's set to WARNS,
  379: the routine should warn with the given arguments.
  380: The output can be set to the string 'undef' if the routine should return the undefined value:
  381: 
  382:   ---
  383:   module: Scalar::Util
  384:   function: blessed
  385: 
  386:   input:
  387:     type: string
  388: 
  389:   output: undef
  390: 
  391: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  392: 
  393: =head3 C<%config> - optional hash of configuration.
  394: 
  395: The current supported variables are
  396: 
  397: =over 4
  398: 
  399: =item * C<close_stdin>
  400: 
  401: Tests should not attempt to read from STDIN (default: 1).
  402: This is ignored on Windows, when never closes STDIN.
  403: 
  404: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  405: 
  406: With this test enabled, the function is expected to die when a NUL byte is passed in.
  407: 
  408: =item * C<test_undef>, test with undefined value (default: 1)
  409: 
  410: =item * C<test_empty>, test with empty strings (default: 1)
  411: 
  412: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  413: 
  414: =item * C<timeout>, ensure tests don't hang (default: 10)
  415: 
  416: Setting this to 0 disables timeout testing.
  417: 
  418: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  419: 
  420: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  421: 
  422: *item * C<test_security>, send some security string based tests (default: 0)
  423: 
  424: =back
  425: 
  426: All values default to C<true>.
  427: 
  428: =head3 C<%accessor> - this is an accessor routine
  429: 
  430:   accessor:
  431:     property: ua
  432:     type: getset
  433: 
  434: Has two mandatory elements:
  435: 
  436: =over 4
  437: 
  438: =item * C<property>
  439: 
  440: The name of the property in the object that the routine controls.
  441: 
  442: =item * C<type>
  443: 
  444: One of C<getter>, C<setter>, C<getset>.
  445: 
  446: =back
  447: 
  448: =head3 C<%transforms> - list of transformations from input sets to output sets
  449: 
  450: Transforms allow you to define how input data should be transformed into output data.
  451: This is useful for testing functions that convert between formats, normalize data,
  452: or apply business logic transformations on a set of data to different set of data.
  453: It takes a list of subsets of the input and output definitions,
  454: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  455: 
  456: =head4 Transform Validation Rules
  457: 
  458: For each transform:
  459: 
  460: =over 4
  461: 
  462: =item 1. Generate test cases using the transform's input schema
  463: 
  464: =item 2. Call the function with those inputs
  465: 
  466: =item 3. Validate the output matches the transform's output schema
  467: 
  468: =item 4. If output has a specific 'value', check exact match
  469: 
  470: =item 5. If output has constraints (min/max), validate within bounds
  471: 
  472: =back

  741: =head2 EDGE CASE GENERATION
  742: 
  743: In addition to purely random fuzz cases, the harness generates
  744: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  745: 
  746: For each constraint, three edge cases are added:
  747: 
  748: =over 4
  749: 
  750: =item * Just inside the allowable range
  751: 
  752: This case should succeed, since it lies strictly within the bounds.
  753: 
  754: =item * Exactly on the boundary
  755: 
  756: This case should succeed, since it meets the constraint exactly.
  757: 
  758: =item * Just outside the boundary
  759: 
  760: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  761: should cause the harness to fail validation or croak.
  762: 
  763: =back
  764: 
  765: Supported constraint types:
  766: 
  767: =over 4
  768: 
  769: =item * C<number>, C<integer>, C<float>
  770: 
  771: Uses numeric values one below, equal to, and one above the boundary.
  772: 
  773: =item * C<string>
  774: 
  775: Uses strings of lengths one below, equal to, and one above the boundary.
  776: 
  777: =item * C<arrayref>
  778: 
  779: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  780: 
  781: =item * C<hashref>
  782: 
  783: Uses hashes with key counts one below, equal to, and one above the
  784: boundary (C<min> = minimum number of keys, C<max> = maximum number
  785: of keys).
  786: 
  787: =item * C<memberof> - arrayref of allowed values for a parameter
  788: 
  789: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  790: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  791: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  792: 
  793:   ---
  794:   input:
  795:     status:
  796:       type: string
  797:       memberof:
  798:         - ok
  799:         - error
  800:         - pending
  801:     level:
  802:       type: integer
  803:       memberof:
  804:         - 1
  805:         - 5
  806:         - 111
  807: 
  808: The generator will automatically create test cases for each allowed value (inside the member list),
  809: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  810: This works for strings, integers, and numbers.
  811: 
  812: =item * C<enum> - synonym of C<memberof>
  813: 
  814: =item * C<boolean> - automatic boundary tests for boolean fields
  815: 
  816:   input:
  817:     flag:
  818:       type: boolean
  819: 
  820: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  821: 
  822: =back
  823: 
  824: These edge cases are inserted automatically, in addition to the random
  825: fuzzing inputs, so each run will reliably probe boundary conditions
  826: without relying solely on randomness.

  986: =head2 Property-Based Testing with Transforms
  987: 
  988: The generator can create property-based tests using L<Test::LectroTest> when the
  989: C<properties> configuration option is enabled.
  990: This provides more comprehensive
  991: testing by automatically generating thousands of test cases and verifying that
  992: mathematical properties hold across all inputs.

  994: =head3 Basic Property-Based Transform Example
  995: 
  996: Here's a complete example testing the C<abs> builtin function:
  997: 
  998: B<t/conf/abs.yml>:
  999: 
 1000:   ---
 1001:   module: builtin
 1002:   function: abs
 1003: 
 1004:   config:
 1005:     test_undef: no
 1006:     test_empty: no
 1007:     test_nuls: no
 1008:     properties:
 1009:       enable: true
 1010:       trials: 1000
 1011: 
 1012:   input:
 1013:     number:
 1014:       type: number
 1015:       position: 0
 1016: 
 1017:   output:
 1018:     type: number
 1019:     min: 0
 1020: 
 1021:   transforms:
 1022:     positive:
 1023:       input:
 1024:         number:
 1025:           type: number
 1026:           min: 0
 1027:       output:
 1028:         type: number
 1029:         min: 0
 1030: 
 1031:     negative:
 1032:       input:
 1033:         number:
 1034:           type: number
 1035:           max: 0
 1036:       output:
 1037:         type: number
 1038:         min: 0
 1039: 
 1040: This configuration:
 1041: 
 1042: =over 4
 1043: 
 1044: =item * Enables property-based testing with 1000 trials per property
 1045: 
 1046: =item * Defines two transforms: one for positive numbers, one for negative
 1047: 
 1048: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1049: 
 1050: =back
 1051:

 1053: 
 1054:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t

 1057: 
 1058: =over 4
 1059: 
 1060: =item * Traditional edge-case tests for boundary conditions
 1061: 
 1062: =item * Random fuzzing with 30 iterations (or as configured)
 1063: 
 1064: =item * Property-based tests that verify the transforms with 1000 trials each
 1065: 
 1066: =back

 1069: 
 1070: The generator automatically detects and tests these properties based on your transform specifications:
 1071: 
 1072: =over 4
 1073: 
 1074: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1075: 
 1076: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1077: 
 1078: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1079:

 1081: 
 1082: =back

 1085: 
 1086:   # For the "positive" transform:

 1089:   - The result is defined
 1090: 
 1091:   # For the "negative" transform:
 1092:   - Given a negative number, abs() returns >= 0
 1093:   - The result is a valid number
 1094:   - The result is defined
