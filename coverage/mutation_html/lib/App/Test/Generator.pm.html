  149: =head1 OVERVIEW
  150: 
  151: This module takes a formal input/output specification for a routine or
  152: method and automatically generates test cases. In effect, it allows you
  153: to easily add comprehensive black-box tests in addition to the more
  154: common white-box tests that are typically written for CPAN modules and other
  155: subroutines.
  156: 
  157: The generated tests combine:
  158: 
  159: =over 4
  160: 
  161: =item * Random fuzzing based on input types
  162: 
  163: =item * Deterministic edge cases for min/max constraints
  164: 
  165: =item * Static corpus tests defined in Perl or YAML
  166: 
  167: =back
  168: 
  169: This approach strengthens your test suite by probing both expected and
  170: unexpected inputs, helping you to catch boundary errors, invalid data
  171: handling, and regressions without manually writing every case.
  172: 
  173: =head1 DESCRIPTION
  174: 
  175: This module implements the logic behind L<fuzz-harness-generator>.
  176: It parses configuration files (fuzz and/or corpus YAML), and
  177: produces a ready-to-run F<.t> test script to run through C<prove>.
  178: 
  179: It reads configuration files in any format,
  180: and optional YAML corpus files.
  181: All of the examples in this documentation are in C<YAML> format,
  182: other formats may not work as they aren't so heavily tested.
  183: It then generates a L<Test::Most>-based fuzzing harness combining:
  184: 
  185: =over 4
  186: 
  187: =item * Randomized fuzzing of inputs (with edge cases)
  188: 
  189: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  190: 
  191: =item * Functional or OO mode (via C<$new>)
  192: 
  193: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  194: 
  195: =back
  196: 
  197: =head1 MUTATION-GUIDED TEST GENERATION
  198: 
  199: C<App::Test::Generator> includes a pipeline that automatically closes the
  200: feedback loop between mutation testing, schema extraction, and fuzz
  201: testing. The goal is that surviving mutants drive the creation of new
  202: tests that kill them on the next run, without manual intervention.
  203: 
  204: =head2 The Pipeline
  205: 
  206:     mutation survivor
  207:         |
  208:         v
  209:     SchemaExtractor extracts the schema for the enclosing sub
  210:         |
  211:         v
  212:     Schema augmented with boundary values from the mutant
  213:         |
  214:         v
  215:     Augmented schema written to t/conf/
  216:         |
  217:         v
  218:     t/fuzz.t picks up the new schema and runs fuzz tests
  219:         |
  220:         v
  221:     Mutation killed on next run
  222: 
  223: =head2 How to Use It
  224: 
  225: The pipeline is driven by three flags passed to
  226: C<bin/app-test-generator-index>, which is invoked automatically by
  227: C<bin/generate-test-dashboard> on each CI push.
  228: 
  229: =head3 Step 1: Generate TODO stubs for all survivors
  230: 
  231:     bin/app-test-generator-index --generate_mutant_tests=t
  232: 
  233: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  234: 
  235: =over 4
  236: 
  237: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  238: boundary value suggestions, environment variable hints, and the
  239: enclosing subroutine name for navigation context.
  240: 
  241: =item * Comment-only hints for LOW difficulty survivors.
  242: 
  243: =back
  244: 
  245: Multiple mutations on the same source line are deduplicated into one
  246: stub. One good test kills all variants on that line.
  247: 
  248: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  249: 
  250:     bin/app-test-generator-index \
  251:         --generate_mutant_tests=t \
  252:         --generate_test=mutant
  253: 
  254: For each NUM_BOUNDARY survivor, calls
  255: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  256: the enclosing subroutine. If the confidence level is sufficient, the
  257: schema is augmented with the boundary value from the mutant (plus one
  258: value either side) and written to C<t/conf/> as a runnable YAML file.
  259: L<t/fuzz.t> picks it up automatically on the next test run.
  260: 
  261: Falls back to a TODO stub if:
  262: 
  263: =over 4
  264: 
  265: =item * SchemaExtractor cannot parse the file
  266: 
  267: =item * The enclosing sub cannot be determined
  268: 
  269: =item * The extracted schema confidence is C<very_low> or C<none>
  270: 
  271: =back
  272: 
  273: =head3 Step 3: Augment existing schemas with survivor boundary values
  274: 
  275:     bin/app-test-generator-index \
  276:         --generate_mutant_tests=t \
  277:         --generate_test=mutant \
  278:         --generate_fuzz
  279: 
  280: Scans C<t/conf/> for existing YAML schema files (hand-written or
  281: previously generated) and writes augmented copies with boundary values
  282: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  283: never modified. Augmented copies are written as
  284: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  285: automatically by C<t/fuzz.t>.
  286: 
  287: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  288: to prevent cascading augmentation. Schemas with no matching survivors
  289: are skipped, with a note if C<--verbose> is active.
  290: 
  291: =head3 Putting It All Together
  292: 
  293: The recommended invocation in C<bin/generate-test-dashboard>
  294: Step 7 runs all three stages together:
  295: 
  296:     bin/app-test-generator-index \
  297:         --generate_mutant_tests=t \
  298:         --generate_test=mutant \
  299:         --generate_fuzz
  300: 
  301: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  302: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  303: to the repository so they accumulate over time as the test suite
  304: improves.

  307: 
  308: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  309: to each extracted schema:
  310: 
  311: =over 4
  312: 
  313: =item * C<high> / C<medium> / C<low> - Schema is used for test generation

  316:

  363: 
  364: The configuration file,
  365: for each set of tests to be produced,

  368: =head2 SCHEMA
  369: 
  370: The schema is split into several sections.
  371: 
  372: =head3 C<%input> - input params with keys => type/optional specs
  373: 
  374: When using named parameters
  375: 
  376:   input:
  377:     name:
  378:       type: string
  379:       optional: false
  380:     age:
  381:       type: integer
  382:       optional: true
  383: 
  384: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  385: See also L<Params::Validate::Strict>.
  386: You can add more custom types using properties.
  387: 
  388: For routines with one unnamed parameter
  389: 
  390:   input:
  391:     type: string
  392: 
  393: For routines with more than one named parameter, use the C<position> keyword.
  394: 
  395:   module: Math::Simple::MinMax
  396:   fuction: max
  397: 
  398:   input:
  399:     left:
  400:       type: number
  401:       position: 0
  402:     right:
  403:       type: number
  404:       position: 1
  405: 
  406:   output:
  407:     type: number
  408: 
  409: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  410: 
  411: =head3 C<%output> - output param types for L<Return::Set> checking
  412: 
  413:   output:
  414:     type: string
  415: 
  416: If the output hash contains the key _STATUS, and if that key is set to DIES,
  417: the routine should die with the given arguments; otherwise, it should live.
  418: If it's set to WARNS,
  419: the routine should warn with the given arguments.
  420: The output can be set to the string 'undef' if the routine should return the undefined value:
  421: 
  422:   ---
  423:   module: Scalar::Util
  424:   function: blessed
  425: 
  426:   input:
  427:     type: string
  428: 
  429:   output: undef
  430: 
  431: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  432: 
  433: =head3 C<%config> - optional hash of configuration.
  434: 
  435: The current supported variables are
  436: 
  437: =over 4
  438: 
  439: =item * C<close_stdin>
  440: 
  441: Tests should not attempt to read from STDIN (default: 1).
  442: This is ignored on Windows, when never closes STDIN.
  443: 
  444: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  445: 
  446: With this test enabled, the function is expected to die when a NUL byte is passed in.
  447: 
  448: =item * C<test_undef>, test with undefined value (default: 1)
  449: 
  450: =item * C<test_empty>, test with empty strings (default: 1)
  451: 
  452: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  453: 
  454: =item * C<timeout>, ensure tests don't hang (default: 10)
  455: 
  456: Setting this to 0 disables timeout testing.
  457: 
  458: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  459: 
  460: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  461: 
  462: *item * C<test_security>, send some security string based tests (default: 0)
  463: 
  464: =back
  465: 
  466: All values default to C<true>.
  467: 
  468: =head3 C<%accessor> - this is an accessor routine
  469: 
  470:   accessor:
  471:     property: ua
  472:     type: getset
  473: 
  474: Has two mandatory elements:
  475: 
  476: =over 4
  477: 
  478: =item * C<property>
  479: 
  480: The name of the property in the object that the routine controls.
  481: 
  482: =item * C<type>
  483: 
  484: One of C<getter>, C<setter>, C<getset>.
  485: 
  486: =back

  488: =head3 C<%transforms> - list of transformations from input sets to output sets
  489: 
  490: Transforms allow you to define how input data should be transformed into output data.
  491: This is useful for testing functions that convert between formats, normalize data,
  492: or apply business logic transformations on a set of data to different set of data.
  493: It takes a list of subsets of the input and output definitions,
  494: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  495: 
  496: =head4 Transform Validation Rules
  497: 
  498: For each transform:
  499: 
  500: =over 4
  501: 
  502: =item 1. Generate test cases using the transform's input schema
  503: 
  504: =item 2. Call the function with those inputs
  505: 
  506: =item 3. Validate the output matches the transform's output schema
  507: 
  508: =item 4. If output has a specific 'value', check exact match
  509: 
  510: =item 5. If output has constraints (min/max), validate within bounds
  511: 
  512: =back

  781: =head2 EDGE CASE GENERATION
  782: 
  783: In addition to purely random fuzz cases, the harness generates
  784: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  785: 
  786: For each constraint, three edge cases are added:
  787: 
  788: =over 4
  789: 
  790: =item * Just inside the allowable range
  791: 
  792: This case should succeed, since it lies strictly within the bounds.
  793: 
  794: =item * Exactly on the boundary
  795: 
  796: This case should succeed, since it meets the constraint exactly.
  797: 
  798: =item * Just outside the boundary
  799: 
  800: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  801: should cause the harness to fail validation or croak.
  802: 
  803: =back
  804: 
  805: Supported constraint types:
  806: 
  807: =over 4
  808: 
  809: =item * C<number>, C<integer>, C<float>
  810: 
  811: Uses numeric values one below, equal to, and one above the boundary.
  812: 
  813: =item * C<string>
  814: 
  815: Uses strings of lengths one below, equal to, and one above the boundary.
  816: 
  817: =item * C<arrayref>
  818: 
  819: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  820: 
  821: =item * C<hashref>
  822: 
  823: Uses hashes with key counts one below, equal to, and one above the
  824: boundary (C<min> = minimum number of keys, C<max> = maximum number
  825: of keys).
  826: 
  827: =item * C<memberof> - arrayref of allowed values for a parameter
  828: 
  829: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  830: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  831: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  832: 
  833:   ---
  834:   input:
  835:     status:
  836:       type: string
  837:       memberof:
  838:         - ok
  839:         - error
  840:         - pending
  841:     level:
  842:       type: integer
  843:       memberof:
  844:         - 1
  845:         - 5
  846:         - 111
  847: 
  848: The generator will automatically create test cases for each allowed value (inside the member list),
  849: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  850: This works for strings, integers, and numbers.
  851: 
  852: =item * C<enum> - synonym of C<memberof>
  853: 
  854: =item * C<boolean> - automatic boundary tests for boolean fields
  855: 
  856:   input:
  857:     flag:
  858:       type: boolean
  859: 
  860: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  861: 
  862: =back
  863: 
  864: These edge cases are inserted automatically, in addition to the random
  865: fuzzing inputs, so each run will reliably probe boundary conditions
  866: without relying solely on randomness.

 1026: =head2 Property-Based Testing with Transforms
 1027: 
 1028: The generator can create property-based tests using L<Test::LectroTest> when the
 1029: C<properties> configuration option is enabled.
 1030: This provides more comprehensive
 1031: testing by automatically generating thousands of test cases and verifying that
 1032: mathematical properties hold across all inputs.

 1035: 
 1036: Here's a complete example testing the C<abs> builtin function:
 1037: 
 1038: B<t/conf/abs.yml>:
 1039: 
 1040:   ---
 1041:   module: builtin
 1042:   function: abs
 1043:

 1045:     test_undef: no
 1046:     test_empty: no
 1047:     test_nuls: no
 1048:     properties:
 1049:       enable: true
 1050:       trials: 1000
 1051: 
 1052:   input:
 1053:     number:
 1054:       type: number
 1055:       position: 0
 1056: 
 1057:   output:
 1058:     type: number
 1059:     min: 0
 1060: 
 1061:   transforms:
 1062:     positive:
 1063:       input:
 1064:         number:
 1065:           type: number
 1066:           min: 0
 1067:       output:
 1068:         type: number
 1069:         min: 0
 1070: 
 1071:     negative:
 1072:       input:
 1073:         number:
 1074:           type: number
 1075:           max: 0
 1076:       output:
 1077:         type: number
 1078:         min: 0
 1079: 
 1080: This configuration:
 1081: 
 1082: =over 4
 1083: 
 1084: =item * Enables property-based testing with 1000 trials per property
 1085: 
 1086: =item * Defines two transforms: one for positive numbers, one for negative
 1087: 
 1088: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1089: 
 1090: =back

 1093: 
 1094:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t

 1097: 
 1098: =over 4
 1099: 
 1100: =item * Traditional edge-case tests for boundary conditions

 1102: =item * Random fuzzing with 30 iterations (or as configured)
 1103: 
 1104: =item * Property-based tests that verify the transforms with 1000 trials each
 1105: 
 1106: =back

 1109: 
 1110: The generator automatically detects and tests these properties based on your transform specifications:
 1111: 
 1112: =over 4
 1113: 
 1114: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1115: 
 1116: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1117: 
 1118: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1119:

 1121: 
 1122: =back

 1125: 
 1126:   # For the "positive" transform:
 1127:   - Given a positive number, abs() returns >= 0

 1129:   - The result is defined
 1130: 
 1131:   # For the "negative" transform:
 1132:   - Given a negative number, abs() returns >= 0
 1133:   - The result is a valid number
 1134:   - The result is defined
