  141: =head1 OVERVIEW
  142: 
  143: This module takes a formal input/output specification for a routine or
  144: method and automatically generates test cases. In effect, it allows you
  145: to easily add comprehensive black-box tests in addition to the more
  146: common white-box tests that are typically written for CPAN modules and other
  147: subroutines.
  148: 
  149: The generated tests combine:
  150: 
  151: =over 4
  152: 
  153: =item * Random fuzzing based on input types
  154: 
  155: =item * Deterministic edge cases for min/max constraints
  156: 
  157: =item * Static corpus tests defined in Perl or YAML
  158: 
  159: =back
  160: 
  161: This approach strengthens your test suite by probing both expected and
  162: unexpected inputs, helping you to catch boundary errors, invalid data
  163: handling, and regressions without manually writing every case.
  164: 
  165: =head1 DESCRIPTION
  166: 
  167: This module implements the logic behind L<fuzz-harness-generator>.
  168: It parses configuration files (fuzz and/or corpus YAML), and
  169: produces a ready-to-run F<.t> test script to run through C<prove>.
  170: 
  171: It reads configuration files in any format,
  172: and optional YAML corpus files.
  173: All of the examples in this documentation are in C<YAML> format,
  174: other formats may not work as they aren't so heavily tested.
  175: It then generates a L<Test::Most>-based fuzzing harness combining:
  176: 
  177: =over 4
  178: 
  179: =item * Randomized fuzzing of inputs (with edge cases)
  180: 
  181: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  182: 
  183: =item * Functional or OO mode (via C<$new>)
  184: 
  185: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  186: 
  187: =back
  188: 
  189: =head1 MUTATION-GUIDED TEST GENERATION
  190: 
  191: C<App::Test::Generator> includes a pipeline that automatically closes the
  192: feedback loop between mutation testing, schema extraction, and fuzz
  193: testing. The goal is that surviving mutants drive the creation of new
  194: tests that kill them on the next run, without manual intervention.
  195: 
  196: =head2 The Pipeline
  197: 
  198:     mutation survivor
  199:         |
  200:         v
  201:     SchemaExtractor extracts the schema for the enclosing sub
  202:         |
  203:         v
  204:     Schema augmented with boundary values from the mutant
  205:         |
  206:         v
  207:     Augmented schema written to t/conf/
  208:         |
  209:         v
  210:     t/fuzz.t picks up the new schema and runs fuzz tests
  211:         |
  212:         v
  213:     Mutation killed on next run
  214: 
  215: =head2 How to Use It
  216: 
  217: The pipeline is driven by three flags passed to
  218: C<bin/app-test-generator-index>, which is invoked automatically by
  219: C<bin/generate-test-dashboard> on each CI push.
  220: 
  221: =head3 Step 1: Generate TODO stubs for all survivors
  222: 
  223:     bin/app-test-generator-index --generate_mutant_tests=t
  224: 
  225: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  226: 
  227: =over 4
  228: 
  229: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  230: boundary value suggestions, environment variable hints, and the
  231: enclosing subroutine name for navigation context.
  232: 
  233: =item * Comment-only hints for LOW difficulty survivors.
  234: 
  235: =back
  236: 
  237: Multiple mutations on the same source line are deduplicated into one
  238: stub. One good test kills all variants on that line.
  239: 
  240: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  241: 
  242:     bin/app-test-generator-index \
  243:         --generate_mutant_tests=t \
  244:         --generate_test=mutant
  245: 
  246: For each NUM_BOUNDARY survivor, calls
  247: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  248: the enclosing subroutine. If the confidence level is sufficient, the
  249: schema is augmented with the boundary value from the mutant (plus one
  250: value either side) and written to C<t/conf/> as a runnable YAML file.
  251: L<t/fuzz.t> picks it up automatically on the next test run.
  252: 
  253: Falls back to a TODO stub if:
  254: 
  255: =over 4
  256: 
  257: =item * SchemaExtractor cannot parse the file
  258: 
  259: =item * The enclosing sub cannot be determined
  260: 
  261: =item * The extracted schema confidence is C<very_low> or C<none>
  262: 
  263: =back
  264: 
  265: =head3 Step 3: Augment existing schemas with survivor boundary values
  266: 
  267:     bin/app-test-generator-index \
  268:         --generate_mutant_tests=t \
  269:         --generate_test=mutant \
  270:         --generate_fuzz
  271: 
  272: Scans C<t/conf/> for existing YAML schema files (hand-written or
  273: previously generated) and writes augmented copies with boundary values
  274: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  275: never modified. Augmented copies are written as
  276: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  277: automatically by C<t/fuzz.t>.
  278: 
  279: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  280: to prevent cascading augmentation. Schemas with no matching survivors
  281: are skipped, with a note if C<--verbose> is active.
  282: 
  283: =head3 Putting It All Together
  284: 
  285: The recommended invocation in C<bin/generate-test-dashboard>
  286: Step 7 runs all three stages together:
  287: 
  288:     bin/app-test-generator-index \
  289:         --generate_mutant_tests=t \
  290:         --generate_test=mutant \
  291:         --generate_fuzz
  292: 
  293: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  294: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  295: to the repository so they accumulate over time as the test suite
  296: improves.

  298: =head2 Confidence Levels
  299: 
  300: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  301: to each extracted schema:
  302: 
  303: =over 4
  304: 
  305: =item * C<high> / C<medium> / C<low> - Schema is used for test generation
  306: 
  307: =item * C<very_low> / C<none> - Falls back to TODO stub
  308: 
  309: =back

  312: be inferred from the source code and its POD documentation. Methods
  313: with explicit parameter validation (L<Params::Validate::Strict>,
  314: L<Params::Get>) or comprehensive POD will produce higher-confidence
  315: schemas.

  355: 
  356: The configuration file,
  357: for each set of tests to be produced,
  358: is a file containing a schema that can be read by L<Config::Abstraction>.
  359: 
  360: =head2 SCHEMA
  361: 
  362: The schema is split into several sections.
  363: 
  364: =head3 C<%input> - input params with keys => type/optional specs
  365: 
  366: When using named parameters
  367: 
  368:   input:
  369:     name:
  370:       type: string
  371:       optional: false
  372:     age:
  373:       type: integer
  374:       optional: true
  375: 
  376: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  377: See also L<Params::Validate::Strict>.
  378: You can add more custom types using properties.
  379: 
  380: For routines with one unnamed parameter
  381: 
  382:   input:
  383:     type: string
  384: 
  385: For routines with more than one named parameter, use the C<position> keyword.
  386: 
  387:   module: Math::Simple::MinMax
  388:   fuction: max
  389: 
  390:   input:
  391:     left:
  392:       type: number
  393:       position: 0
  394:     right:
  395:       type: number
  396:       position: 1
  397: 
  398:   output:
  399:     type: number
  400: 
  401: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  402: 
  403: =head3 C<%output> - output param types for L<Return::Set> checking
  404: 
  405:   output:
  406:     type: string
  407: 
  408: If the output hash contains the key _STATUS, and if that key is set to DIES,
  409: the routine should die with the given arguments; otherwise, it should live.
  410: If it's set to WARNS,
  411: the routine should warn with the given arguments.
  412: The output can be set to the string 'undef' if the routine should return the undefined value:
  413: 
  414:   ---
  415:   module: Scalar::Util
  416:   function: blessed
  417: 
  418:   input:
  419:     type: string
  420: 
  421:   output: undef
  422: 
  423: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  424: 
  425: =head3 C<%config> - optional hash of configuration.
  426: 
  427: The current supported variables are
  428: 
  429: =over 4
  430: 
  431: =item * C<close_stdin>
  432: 
  433: Tests should not attempt to read from STDIN (default: 1).
  434: This is ignored on Windows, when never closes STDIN.
  435: 
  436: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  437: 
  438: With this test enabled, the function is expected to die when a NUL byte is passed in.
  439: 
  440: =item * C<test_undef>, test with undefined value (default: 1)
  441: 
  442: =item * C<test_empty>, test with empty strings (default: 1)
  443: 
  444: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  445: 
  446: =item * C<timeout>, ensure tests don't hang (default: 10)
  447: 
  448: Setting this to 0 disables timeout testing.
  449: 
  450: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  451: 
  452: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  453: 
  454: *item * C<test_security>, send some security string based tests (default: 0)
  455: 
  456: =back
  457: 
  458: All values default to C<true>.
  459: 
  460: =head3 C<%accessor> - this is an accessor routine
  461: 
  462:   accessor:
  463:     property: ua
  464:     type: getset
  465: 
  466: Has two mandatory elements:
  467: 
  468: =over 4
  469: 
  470: =item * C<property>
  471: 
  472: The name of the property in the object that the routine controls.
  473: 
  474: =item * C<type>
  475: 
  476: One of C<getter>, C<setter>, C<getset>.
  477: 
  478: =back
  479: 
  480: =head3 C<%transforms> - list of transformations from input sets to output sets
  481: 
  482: Transforms allow you to define how input data should be transformed into output data.
  483: This is useful for testing functions that convert between formats, normalize data,
  484: or apply business logic transformations on a set of data to different set of data.
  485: It takes a list of subsets of the input and output definitions,
  486: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  487: 
  488: =head4 Transform Validation Rules
  489: 
  490: For each transform:
  491: 
  492: =over 4
  493: 
  494: =item 1. Generate test cases using the transform's input schema
  495: 
  496: =item 2. Call the function with those inputs
  497: 
  498: =item 3. Validate the output matches the transform's output schema
  499: 
  500: =item 4. If output has a specific 'value', check exact match
  501: 
  502: =item 5. If output has constraints (min/max), validate within bounds
  503: 
  504: =back

  773: =head2 EDGE CASE GENERATION
  774: 
  775: In addition to purely random fuzz cases, the harness generates
  776: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  777: 
  778: For each constraint, three edge cases are added:
  779: 
  780: =over 4
  781: 
  782: =item * Just inside the allowable range
  783: 
  784: This case should succeed, since it lies strictly within the bounds.
  785: 
  786: =item * Exactly on the boundary
  787: 
  788: This case should succeed, since it meets the constraint exactly.
  789: 
  790: =item * Just outside the boundary
  791: 
  792: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  793: should cause the harness to fail validation or croak.
  794: 
  795: =back
  796: 
  797: Supported constraint types:
  798: 
  799: =over 4
  800: 
  801: =item * C<number>, C<integer>, C<float>
  802: 
  803: Uses numeric values one below, equal to, and one above the boundary.
  804: 
  805: =item * C<string>
  806: 
  807: Uses strings of lengths one below, equal to, and one above the boundary.
  808: 
  809: =item * C<arrayref>
  810: 
  811: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  812: 
  813: =item * C<hashref>
  814: 
  815: Uses hashes with key counts one below, equal to, and one above the
  816: boundary (C<min> = minimum number of keys, C<max> = maximum number
  817: of keys).
  818: 
  819: =item * C<memberof> - arrayref of allowed values for a parameter
  820: 
  821: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  822: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  823: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  824: 
  825:   ---
  826:   input:
  827:     status:
  828:       type: string
  829:       memberof:
  830:         - ok
  831:         - error
  832:         - pending
  833:     level:
  834:       type: integer
  835:       memberof:
  836:         - 1
  837:         - 5
  838:         - 111
  839: 
  840: The generator will automatically create test cases for each allowed value (inside the member list),
  841: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  842: This works for strings, integers, and numbers.
  843: 
  844: =item * C<enum> - synonym of C<memberof>
  845: 
  846: =item * C<boolean> - automatic boundary tests for boolean fields
  847: 
  848:   input:
  849:     flag:
  850:       type: boolean
  851: 
  852: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  853: 
  854: =back
  855: 
  856: These edge cases are inserted automatically, in addition to the random
  857: fuzzing inputs, so each run will reliably probe boundary conditions
  858: without relying solely on randomness.

 1019: 
 1020: The generator can create property-based tests using L<Test::LectroTest> when the
 1021: C<properties> configuration option is enabled.

 1023: testing by automatically generating thousands of test cases and verifying that

 1048: 
 1049:   output:

 1051:     min: 0
 1052:

 1056:         number:

 1058:           min: 0
 1059:       output:
 1060:         type: number
 1061:         min: 0
 1062: 
 1063:     negative:
