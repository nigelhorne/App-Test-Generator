  217: =head1 OVERVIEW
  218: 
  219: This module takes a formal input/output specification for a routine or
  220: method and automatically generates test cases. In effect, it allows you
  221: to easily add comprehensive black-box tests in addition to the more
  222: common white-box tests that are typically written for CPAN modules and other
  223: subroutines.
  224: 
  225: The generated tests combine:
  226: 
  227: =over 4
  228: 
  229: =item * Random fuzzing based on input types
  230: 
  231: =item * Deterministic edge cases for min/max constraints
  232: 
  233: =item * Static corpus tests defined in Perl or YAML
  234: 
  235: =back
  236: 
  237: This approach strengthens your test suite by probing both expected and
  238: unexpected inputs, helping you to catch boundary errors, invalid data
  239: handling, and regressions without manually writing every case.
  240: 
  241: =head1 TOOLS
  242: 
  243: The distribution ships the following command-line tools:
  244: 
  245: =over 4
  246: 
  247: =item * L<extract-schemas> â€” heuristically extract YAML parameter schemas from a C<.pm> file, with optional coverage-guided fuzzing (C<--fuzz>) and corpus minimization (C<--minimize-corpus>).
  248: 
  249: =item * L<fuzz-harness-generator> â€” generate a C<Test::Most> fuzzing harness from a YAML schema.
  250: 
  251: =item * L<pod-example-tester> â€” generate a C<Test::Most> round-trip test file from a module's POD code examples. Annotated examples (C<# returns value> / C<< # => value >>) get C<is()> assertions; unannotated verbatim blocks are wrapped in C<eval{}> and checked for no exception.
  252: 
  253: =item * L<test-generator-mutate> â€” run mutation testing against a module's test suite.
  254: 
  255: =item * L<test-generator-index> â€” generate the HTML test-quality dashboard, combining Devel::Cover statement/branch data, LCSAJ path coverage, mutation results, and CPAN Testers failure analysis. For each CPAN Testers FAIL report, also writes a self-contained shell script (C<cover_html/reproduce/reproduce-GUID.sh>) that pins every installed module at its exact failing version, enabling local reproduction of the failure environment.
  256: 
  257: =back

  259: =head1 DESCRIPTION
  260: 
  261: This module implements the logic behind L<fuzz-harness-generator>.
  262: It parses configuration files (fuzz and/or corpus YAML), and
  263: produces a ready-to-run F<.t> test script to run through C<prove>.
  264: 
  265: It reads configuration files in any format,
  266: and optional YAML corpus files.
  267: All of the examples in this documentation are in C<YAML> format,
  268: other formats may not work as they aren't so heavily tested.
  269: It then generates a L<Test::Most>-based fuzzing harness combining:
  270: 
  271: =over 4
  272: 
  273: =item * Randomized fuzzing of inputs (with edge cases)
  274: 
  275: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  276: 
  277: =item * Functional or OO mode (via C<$new>)
  278: 
  279: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  280: 
  281: =back

  283: =head1 MUTATION-GUIDED TEST GENERATION
  284: 
  285: C<App::Test::Generator> includes a pipeline that automatically closes the
  286: feedback loop between mutation testing, schema extraction, and fuzz
  287: testing. The goal is that surviving mutants drive the creation of new
  288: tests that kill them on the next run, without manual intervention.
  289: 
  290: =head2 The Pipeline
  291: 
  292:     mutation survivor
  293:         |
  294:         v
  295:     SchemaExtractor extracts the schema for the enclosing sub
  296:         |
  297:         v
  298:     Schema augmented with boundary values from the mutant
  299:         |
  300:         v
  301:     Augmented schema written to t/conf/
  302:         |
  303:         v
  304:     t/fuzz.t picks up the new schema and runs fuzz tests
  305:         |
  306:         v
  307:     Mutation killed on next run
  308: 
  309: =head2 How to Use It
  310: 
  311: The pipeline is driven by three flags passed to
  312: C<bin/test-generator-index>, which is invoked automatically by
  313: C<bin/generate-test-dashboard> on each CI push.
  314: 
  315: =head3 Step 1: Generate TODO stubs for all survivors
  316: 
  317:     bin/test-generator-index --generate_mutant_tests=t
  318: 
  319: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  320: 
  321: =over 4
  322: 
  323: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  324: boundary value suggestions, environment variable hints, and the
  325: enclosing subroutine name for navigation context.
  326: 
  327: =item * Comment-only hints for LOW difficulty survivors.
  328: 
  329: =back
  330: 
  331: Multiple mutations on the same source line are deduplicated into one
  332: stub. One good test kills all variants on that line.
  333: 
  334: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  335: 
  336:     bin/test-generator-index \
  337:         --generate_mutant_tests=t \
  338:         --generate_test=mutant
  339: 
  340: For each NUM_BOUNDARY survivor, calls
  341: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  342: the enclosing subroutine. If the confidence level is sufficient, the
  343: schema is augmented with the boundary value from the mutant (plus one
  344: value either side) and written to C<t/conf/> as a runnable YAML file.
  345: L<t/fuzz.t> picks it up automatically on the next test run.
  346: 
  347: Falls back to a TODO stub if:
  348: 
  349: =over 4
  350: 
  351: =item * SchemaExtractor cannot parse the file
  352: 
  353: =item * The enclosing sub cannot be determined
  354: 
  355: =item * The extracted schema confidence is C<very_low> or C<none>
  356: 
  357: =back
  358: 
  359: =head3 Step 3: Augment existing schemas with survivor boundary values
  360: 
  361:     bin/test-generator-index \
  362:         --generate_mutant_tests=t \
  363:         --generate_test=mutant \
  364:         --generate_fuzz
  365: 
  366: Scans C<t/conf/> for existing YAML schema files (hand-written or
  367: previously generated) and writes augmented copies with boundary values
  368: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  369: never modified. Augmented copies are written as
  370: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  371: automatically by C<t/fuzz.t>.
  372: 
  373: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  374: to prevent cascading augmentation. Schemas with no matching survivors
  375: are skipped, with a note if C<--verbose> is active.
  376: 
  377: =head3 Putting It All Together
  378: 
  379: The recommended invocation in C<bin/generate-test-dashboard>
  380: Step 7 runs all three stages together:
  381: 
  382:     bin/test-generator-index \
  383:         --generate_mutant_tests=t \
  384:         --generate_test=mutant \
  385:         --generate_fuzz
  386: 
  387: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  388: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  389: to the repository so they accumulate over time as the test suite
  390: improves.

  392: =head2 Confidence Levels
  393: 
  394: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  395: to each extracted schema:
  396: 
  397: =over 4
  398: 
  399: =item * C<high> / C<medium> / C<low> - Schema is used for test generation
  400:

  407: with explicit parameter validation (L<Params::Validate::Strict>,
  408: L<Params::Get>) or comprehensive POD will produce higher-confidence
  409: schemas.

  449: 
  450: The configuration file,
  451: for each set of tests to be produced,
  452: is a file containing a schema that can be read by L<Config::Abstraction>.

  454: =head2 SCHEMA
  455: 
  456: The schema is split into several sections.
  457: 
  458: =head3 C<%input> - input params with keys => type/optional specs
  459: 
  460: When using named parameters
  461: 
  462:   input:
  463:     name:
  464:       type: string
  465:       optional: false
  466:     age:
  467:       type: integer
  468:       optional: true
  469: 
  470: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  471: See also L<Params::Validate::Strict>.
  472: You can add more custom types using properties.
  473: 
  474: For routines with one unnamed parameter
  475: 
  476:   input:
  477:     type: string
  478: 
  479: For routines with more than one named parameter, use the C<position> keyword.
  480: 
  481:   module: Math::Simple::MinMax
  482:   fuction: max
  483: 
  484:   input:
  485:     left:
  486:       type: number
  487:       position: 0
  488:     right:
  489:       type: number
  490:       position: 1
  491: 
  492:   output:
  493:     type: number
  494: 
  495: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  496: 
  497: =head3 C<%output> - output param types for L<Return::Set> checking
  498: 
  499:   output:
  500:     type: string
  501: 
  502: If the output hash contains the key _STATUS, and if that key is set to DIES,
  503: the routine should die with the given arguments; otherwise, it should live.
  504: If it's set to WARNS,
  505: the routine should warn with the given arguments.
  506: The output can be set to the string 'undef' if the routine should return the undefined value:
  507: 
  508:   ---
  509:   module: Scalar::Util
  510:   function: blessed
  511: 
  512:   input:
  513:     type: string
  514: 
  515:   output: undef
  516: 
  517: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  518: 
  519: For methods that return a list (rather than a reference), use C<type: array>.
  520: The generated test captures the result in list context and validates it as an
  521: arrayref, which requires L<Test::Returns> 0.03 or later:
  522: 
  523:   output:
  524:     type: array
  525: 
  526: =head3 C<%config> - optional hash of configuration.
  527: 
  528: The current supported variables are
  529: 
  530: =over 4
  531: 
  532: =item * C<close_stdin>
  533: 
  534: Tests should not attempt to read from STDIN (default: 1).
  535: This is ignored on Windows, when never closes STDIN.
  536: 
  537: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  538: 
  539: With this test enabled, the function is expected to die when a NUL byte is passed in.
  540: 
  541: =item * C<test_undef>, test with undefined value (default: 1)
  542: 
  543: =item * C<test_empty>, test with empty strings (default: 1)
  544: 
  545: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  546: 
  547: =item * C<timeout>, ensure tests don't hang (default: 10)
  548: 
  549: Setting this to 0 disables timeout testing.
  550: 
  551: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  552: 
  553: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  554: 
  555: *item * C<test_security>, send some security string based tests (default: 0)
  556: 
  557: =back
  558: 
  559: All values default to C<true>.
  560: 
  561: =head3 C<%accessor> - this is an accessor routine
  562: 
  563:   accessor:
  564:     property: ua
  565:     type: getset
  566: 
  567: Has two mandatory elements:
  568: 
  569: =over 4
  570: 
  571: =item * C<property>
  572: 
  573: The name of the property in the object that the routine controls.
  574: 
  575: =item * C<type>
  576: 
  577: One of C<getter>, C<setter>, C<getset>.
  578: 
  579: =back
  580: 
  581: =head3 C<%transforms> - list of transformations from input sets to output sets
  582: 
  583: Transforms allow you to define how input data should be transformed into output data.
  584: This is useful for testing functions that convert between formats, normalize data,
  585: or apply business logic transformations on a set of data to different set of data.
  586: It takes a list of subsets of the input and output definitions,
  587: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  588: 
  589: =head4 Transform Validation Rules
  590: 
  591: For each transform:
  592: 
  593: =over 4
  594: 
  595: =item 1. Generate test cases using the transform's input schema
  596: 
  597: =item 2. Call the function with those inputs
  598: 
  599: =item 3. Validate the output matches the transform's output schema
  600: 
  601: =item 4. If output has a specific 'value', check exact match
  602: 
  603: =item 5. If output has constraints (min/max), validate within bounds
  604: 
  605: =back

  874: =head2 EDGE CASE GENERATION
  875: 
  876: In addition to purely random fuzz cases, the harness generates
  877: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  878: 
  879: For each constraint, three edge cases are added:
  880: 
  881: =over 4
  882: 
  883: =item * Just inside the allowable range
  884: 
  885: This case should succeed, since it lies strictly within the bounds.
  886: 
  887: =item * Exactly on the boundary
  888: 
  889: This case should succeed, since it meets the constraint exactly.
  890: 
  891: =item * Just outside the boundary
  892: 
  893: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  894: should cause the harness to fail validation or croak.
  895: 
  896: =back
  897: 
  898: Supported constraint types:
  899: 
  900: =over 4
  901: 
  902: =item * C<number>, C<integer>, C<float>
  903: 
  904: Uses numeric values one below, equal to, and one above the boundary.
  905: 
  906: =item * C<string>
  907: 
  908: Uses strings of lengths one below, equal to, and one above the boundary.
  909: 
  910: =item * C<arrayref>
  911: 
  912: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  913: 
  914: =item * C<hashref>
  915: 
  916: Uses hashes with key counts one below, equal to, and one above the
  917: boundary (C<min> = minimum number of keys, C<max> = maximum number
  918: of keys).
  919: 
  920: =item * C<memberof> - arrayref of allowed values for a parameter
  921: 
  922: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  923: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  924: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  925: 
  926:   ---
  927:   input:
  928:     status:
  929:       type: string
  930:       memberof:
  931:         - ok
  932:         - error
  933:         - pending
  934:     level:
  935:       type: integer
  936:       memberof:
  937:         - 1
  938:         - 5
  939:         - 111
  940: 
  941: The generator will automatically create test cases for each allowed value (inside the member list),
  942: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  943: This works for strings, integers, and numbers.
  944: 
  945: =item * C<enum> - synonym of C<memberof>
  946: 
  947: =item * C<boolean> - automatic boundary tests for boolean fields
  948: 
  949:   input:
  950:     flag:
  951:       type: boolean
  952: 
  953: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  954: 
  955: =back
  956: 
  957: These edge cases are inserted automatically, in addition to the random
  958: fuzzing inputs, so each run will reliably probe boundary conditions
  959: without relying solely on randomness.

 1119: =head2 Property-Based Testing with Transforms
 1120: 
 1121: The generator can create property-based tests using L<Test::LectroTest> when the
 1122: C<properties> configuration option is enabled.
 1123: This provides more comprehensive
 1124: testing by automatically generating thousands of test cases and verifying that
 1125: mathematical properties hold across all inputs.

 1128: 
 1129: Here's a complete example testing the C<abs> builtin function:
 1130: 
 1131: B<t/conf/abs.yml>:
 1132: 
 1133:   ---
 1134:   module: builtin

 1136:

 1139:     test_empty: no
 1140:     test_nuls: no
 1141:     properties:
 1142:       enable: true
 1143:       trials: 1000
 1144: 
 1145:   input:
 1146:     number:
 1147:       type: number
 1148:       position: 0
 1149: 
 1150:   output:
 1151:     type: number
 1152:     min: 0
 1153: 
 1154:   transforms:
 1155:     positive:
 1156:       input:
 1157:         number:
 1158:           type: number
 1159:           min: 0
 1160:       output:
 1161:         type: number
 1162:         min: 0
 1163: 
 1164:     negative:
 1165:       input:
 1166:         number:
 1167:           type: number
 1168:           max: 0
 1169:       output:
 1170:         type: number
 1171:         min: 0
 1172:

 1174: 
 1175: =over 4
 1176: 
 1177: =item * Enables property-based testing with 1000 trials per property
 1178:

 1180: 
 1181: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1182: 
 1183: =back

 1186: 
 1187:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t

 1190: 
 1191: =over 4
 1192: 
 1193: =item * Traditional edge-case tests for boundary conditions
 1194:

 1196: 
 1197: =item * Property-based tests that verify the transforms with 1000 trials each
 1198: 
 1199: =back

 1202: 
 1203: The generator automatically detects and tests these properties based on your transform specifications:
 1204: 
 1205: =over 4
 1206: 
 1207: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1208: 
 1209: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1210: 
 1211: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1212: 
 1213: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
 1214: 
 1215: =back

 1218: 
 1219:   # For the "positive" transform:
 1220:   - Given a positive number, abs() returns >= 0

 1222:   - The result is defined
 1223: 
 1224:   # For the "negative" transform:
 1225:   - Given a negative number, abs() returns >= 0
 1226:   - The result is a valid number
 1227:   - The result is defined

 1302:
