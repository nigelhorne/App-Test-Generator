  218: =head1 OVERVIEW
  219: 
  220: This module takes a formal input/output specification for a routine or
  221: method and automatically generates test cases. In effect, it allows you
  222: to easily add comprehensive black-box tests in addition to the more
  223: common white-box tests that are typically written for CPAN modules and other
  224: subroutines.
  225: 
  226: The generated tests combine:
  227: 
  228: =over 4
  229: 
  230: =item * Random fuzzing based on input types
  231: 
  232: =item * Deterministic edge cases for min/max constraints
  233: 
  234: =item * Static corpus tests defined in Perl or YAML
  235: 
  236: =back
  237: 
  238: This approach strengthens your test suite by probing both expected and
  239: unexpected inputs, helping you to catch boundary errors, invalid data
  240: handling, and regressions without manually writing every case.
  241: 
  242: =head1 TOOLS
  243: 
  244: The distribution ships the following command-line tools:
  245: 
  246: =over 4
  247: 
  248: =item * L<extract-schemas> - heuristically extract YAML parameter schemas from a C<.pm> file, with optional coverage-guided fuzzing (C<--fuzz>) and corpus minimization (C<--minimize-corpus>).
  249: 
  250: =item * L<fuzz-harness-generator> - generate a C<Test::Most> fuzzing harness from a YAML schema.
  251: 
  252: =item * L<pod-example-tester> - generate a C<Test::Most> round-trip test file from a module's POD code examples. Annotated examples (C<# returns value> / C<< # => value >>) get C<is()> assertions; unannotated verbatim blocks are wrapped in C<eval{}> and checked for no exception.
  253: 
  254: =item * L<test-generator-mutate> - run mutation testing against a module's test suite.
  255: 
  256: =item * L<test-generator-index> - generate the HTML test-quality dashboard, combining Devel::Cover statement/branch data, LCSAJ path coverage, mutation results, and CPAN Testers failure analysis. For each CPAN Testers FAIL report, also writes a self-contained shell script (C<cover_html/reproduce/reproduce-GUID.sh>) that pins every installed module at its exact failing version, enabling local reproduction of the failure environment.
  257: 
  258: =back
  259: 
  260: =head1 DESCRIPTION
  261: 
  262: This module implements the logic behind L<fuzz-harness-generator>.
  263: It parses configuration files (fuzz and/or corpus YAML), and
  264: produces a ready-to-run F<.t> test script to run through C<prove>.
  265: 
  266: It reads configuration files in any format,
  267: and optional YAML corpus files.
  268: All of the examples in this documentation are in C<YAML> format,
  269: other formats may not work as they aren't so heavily tested.
  270: It then generates a L<Test::Most>-based fuzzing harness combining:
  271: 
  272: =over 4
  273: 
  274: =item * Randomized fuzzing of inputs (with edge cases)
  275: 
  276: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  277: 
  278: =item * Functional or OO mode (via C<$new>)
  279: 
  280: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  281: 
  282: =back
  283: 
  284: =head1 MUTATION-GUIDED TEST GENERATION
  285: 
  286: C<App::Test::Generator> includes a pipeline that automatically closes the
  287: feedback loop between mutation testing, schema extraction, and fuzz
  288: testing. The goal is that surviving mutants drive the creation of new
  289: tests that kill them on the next run, without manual intervention.
  290: 
  291: =head2 The Pipeline
  292: 
  293:     mutation survivor
  294:         |
  295:         v
  296:     SchemaExtractor extracts the schema for the enclosing sub
  297:         |
  298:         v
  299:     Schema augmented with boundary values from the mutant
  300:         |
  301:         v
  302:     Augmented schema written to t/conf/
  303:         |
  304:         v
  305:     t/fuzz.t picks up the new schema and runs fuzz tests
  306:         |
  307:         v
  308:     Mutation killed on next run
  309: 
  310: =head2 How to Use It
  311: 
  312: The pipeline is driven by three flags passed to
  313: C<bin/test-generator-index>, which is invoked automatically by
  314: C<bin/generate-test-dashboard> on each CI push.
  315: 
  316: =head3 Step 1: Generate TODO stubs for all survivors
  317: 
  318:     bin/test-generator-index --generate_mutant_tests=t
  319: 
  320: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  321: 
  322: =over 4
  323: 
  324: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  325: boundary value suggestions, environment variable hints, and the
  326: enclosing subroutine name for navigation context.
  327: 
  328: =item * Comment-only hints for LOW difficulty survivors.
  329: 
  330: =back
  331: 
  332: Multiple mutations on the same source line are deduplicated into one
  333: stub. One good test kills all variants on that line.
  334: 
  335: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  336: 
  337:     bin/test-generator-index \
  338:         --generate_mutant_tests=t \
  339:         --generate_test=mutant
  340: 
  341: For each NUM_BOUNDARY survivor, calls
  342: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  343: the enclosing subroutine. If the confidence level is sufficient, the
  344: schema is augmented with the boundary value from the mutant (plus one
  345: value either side) and written to C<t/conf/> as a runnable YAML file.
  346: L<t/fuzz.t> picks it up automatically on the next test run.
  347: 
  348: Falls back to a TODO stub if:
  349: 
  350: =over 4
  351: 
  352: =item * SchemaExtractor cannot parse the file
  353: 
  354: =item * The enclosing sub cannot be determined
  355: 
  356: =item * The extracted schema confidence is C<very_low> or C<none>
  357: 
  358: =back
  359: 
  360: =head3 Step 3: Augment existing schemas with survivor boundary values
  361: 
  362:     bin/test-generator-index \
  363:         --generate_mutant_tests=t \
  364:         --generate_test=mutant \
  365:         --generate_fuzz
  366: 
  367: Scans C<t/conf/> for existing YAML schema files (hand-written or
  368: previously generated) and writes augmented copies with boundary values
  369: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  370: never modified. Augmented copies are written as
  371: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  372: automatically by C<t/fuzz.t>.
  373: 
  374: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  375: to prevent cascading augmentation. Schemas with no matching survivors
  376: are skipped, with a note if C<--verbose> is active.
  377: 
  378: =head3 Putting It All Together
  379: 
  380: The recommended invocation in C<bin/generate-test-dashboard>
  381: Step 7 runs all three stages together:
  382: 
  383:     bin/test-generator-index \
  384:         --generate_mutant_tests=t \
  385:         --generate_test=mutant \
  386:         --generate_fuzz
  387: 
  388: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  389: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  390: to the repository so they accumulate over time as the test suite
  391: improves.

  393: =head2 Confidence Levels
  394: 
  395: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  396: to each extracted schema:
  397: 
  398: =over 4
  399: 
  400: =item * C<high> / C<medium> / C<low> - Schema is used for test generation
  401: 
  402: =item * C<very_low> / C<none> - Falls back to TODO stub
  403: 
  404: =back
  405: 
  406: Confidence is based on how much type and constraint information could
  407: be inferred from the source code and its POD documentation. Methods
  408: with explicit parameter validation (L<Params::Validate::Strict>,
  409: L<Params::Get>) or comprehensive POD will produce higher-confidence
  410: schemas.

  449: =head1 CONFIGURATION
  450: 
  451: The configuration file,
  452: for each set of tests to be produced,
  453: is a file containing a schema that can be read by L<Config::Abstraction>.
  454: 
  455: =head2 SCHEMA
  456: 
  457: The schema is split into several sections.
  458: 
  459: =head3 C<%input> - input params with keys => type/optional specs
  460: 
  461: When using named parameters
  462: 
  463:   input:
  464:     name:
  465:       type: string
  466:       optional: false
  467:     age:
  468:       type: integer
  469:       optional: true
  470: 
  471: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  472: See also L<Params::Validate::Strict>.
  473: You can add more custom types using properties.
  474: 
  475: For routines with one unnamed parameter
  476: 
  477:   input:
  478:     type: string
  479: 
  480: For routines with more than one named parameter, use the C<position> keyword.
  481: 
  482:   module: Math::Simple::MinMax
  483:   fuction: max
  484: 
  485:   input:
  486:     left:
  487:       type: number
  488:       position: 0
  489:     right:
  490:       type: number
  491:       position: 1
  492: 
  493:   output:
  494:     type: number
  495: 
  496: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  497: 
  498: =head3 C<%output> - output param types for L<Return::Set> checking
  499: 
  500:   output:
  501:     type: string
  502: 
  503: If the output hash contains the key _STATUS, and if that key is set to DIES,
  504: the routine should die with the given arguments; otherwise, it should live.
  505: If it's set to WARNS,
  506: the routine should warn with the given arguments.
  507: The output can be set to the string 'undef' if the routine should return the undefined value:
  508: 
  509:   ---
  510:   module: Scalar::Util
  511:   function: blessed
  512: 
  513:   input:
  514:     type: string
  515: 
  516:   output: undef
  517: 
  518: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  519: 
  520: For methods that return a list (rather than a reference), use C<type: array>.
  521: The generated test captures the result in list context and validates it as an
  522: arrayref, which requires L<Test::Returns> 0.03 or later:
  523: 
  524:   output:
  525:     type: array
  526: 
  527: =head3 C<%config> - optional hash of configuration.
  528: 
  529: The current supported variables are
  530: 
  531: =over 4
  532: 
  533: =item * C<close_stdin>
  534: 
  535: Tests should not attempt to read from STDIN (default: 1).
  536: This is ignored on Windows, when never closes STDIN.
  537: 
  538: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  539: 
  540: With this test enabled, the function is expected to die when a NUL byte is passed in.
  541: 
  542: =item * C<test_undef>, test with undefined value (default: 1)
  543: 
  544: =item * C<test_empty>, test with empty strings (default: 1)
  545: 
  546: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  547: 
  548: =item * C<timeout>, ensure tests don't hang (default: 10)
  549: 
  550: Setting this to 0 disables timeout testing.
  551: 
  552: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  553: 
  554: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  555: 
  556: *item * C<test_security>, send some security string based tests (default: 0)
  557: 
  558: =back
  559: 
  560: All values default to C<true>.
  561: 
  562: =head3 C<%accessor> - this is an accessor routine
  563: 
  564:   accessor:
  565:     property: ua
  566:     type: getset
  567: 
  568: Has two mandatory elements:
  569: 
  570: =over 4
  571: 
  572: =item * C<property>
  573: 
  574: The name of the property in the object that the routine controls.
  575: 
  576: =item * C<type>
  577: 
  578: One of C<getter>, C<setter>, C<getset>.
  579: 
  580: =back
  581: 
  582: =head3 C<%transforms> - list of transformations from input sets to output sets
  583: 
  584: Transforms allow you to define how input data should be transformed into output data.
  585: This is useful for testing functions that convert between formats, normalize data,
  586: or apply business logic transformations on a set of data to different set of data.
  587: It takes a list of subsets of the input and output definitions,
  588: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  589: 
  590: =head4 Transform Validation Rules
  591: 
  592: For each transform:
  593: 
  594: =over 4
  595: 
  596: =item 1. Generate test cases using the transform's input schema
  597: 
  598: =item 2. Call the function with those inputs
  599: 
  600: =item 3. Validate the output matches the transform's output schema
  601: 
  602: =item 4. If output has a specific 'value', check exact match
  603: 
  604: =item 5. If output has constraints (min/max), validate within bounds
  605: 
  606: =back

  875: =head2 EDGE CASE GENERATION
  876: 
  877: In addition to purely random fuzz cases, the harness generates
  878: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  879: 
  880: For each constraint, three edge cases are added:
  881: 
  882: =over 4
  883: 
  884: =item * Just inside the allowable range
  885: 
  886: This case should succeed, since it lies strictly within the bounds.
  887: 
  888: =item * Exactly on the boundary
  889: 
  890: This case should succeed, since it meets the constraint exactly.
  891: 
  892: =item * Just outside the boundary
  893: 
  894: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  895: should cause the harness to fail validation or croak.
  896: 
  897: =back
  898: 
  899: Supported constraint types:
  900: 
  901: =over 4
  902: 
  903: =item * C<number>, C<integer>, C<float>
  904: 
  905: Uses numeric values one below, equal to, and one above the boundary.
  906: 
  907: =item * C<string>
  908: 
  909: Uses strings of lengths one below, equal to, and one above the boundary.
  910: 
  911: =item * C<arrayref>
  912: 
  913: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  914: 
  915: =item * C<hashref>
  916: 
  917: Uses hashes with key counts one below, equal to, and one above the
  918: boundary (C<min> = minimum number of keys, C<max> = maximum number
  919: of keys).
  920: 
  921: =item * C<memberof> - arrayref of allowed values for a parameter
  922: 
  923: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  924: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  925: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  926: 
  927:   ---
  928:   input:
  929:     status:
  930:       type: string
  931:       memberof:
  932:         - ok
  933:         - error
  934:         - pending
  935:     level:
  936:       type: integer
  937:       memberof:
  938:         - 1
  939:         - 5
  940:         - 111
  941: 
  942: The generator will automatically create test cases for each allowed value (inside the member list),
  943: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  944: This works for strings, integers, and numbers.
  945: 
  946: =item * C<enum> - synonym of C<memberof>
  947: 
  948: =item * C<boolean> - automatic boundary tests for boolean fields
  949: 
  950:   input:
  951:     flag:
  952:       type: boolean
  953: 
  954: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  955: 
  956: =back
  957: 
  958: These edge cases are inserted automatically, in addition to the random
  959: fuzzing inputs, so each run will reliably probe boundary conditions
  960: without relying solely on randomness.

 1120: =head2 Property-Based Testing with Transforms
 1121: 
 1122: The generator can create property-based tests using L<Test::LectroTest> when the
 1123: C<properties> configuration option is enabled.
 1124: This provides more comprehensive
 1125: testing by automatically generating thousands of test cases and verifying that
 1126: mathematical properties hold across all inputs.

 1129: 
 1130: Here's a complete example testing the C<abs> builtin function:
 1131: 
 1132: B<t/conf/abs.yml>:
 1133: 
 1134:   ---
 1135:   module: builtin
 1136:   function: abs
 1137: 
 1138:   config:
 1139:     test_undef: no
 1140:     test_empty: no
 1141:     test_nuls: no
 1142:     properties:
 1143:       enable: true
 1144:       trials: 1000
 1145: 
 1146:   input:
 1147:     number:
 1148:       type: number
 1149:       position: 0
 1150: 
 1151:   output:
 1152:     type: number
 1153:     min: 0
 1154: 
 1155:   transforms:
 1156:     positive:
 1157:       input:
 1158:         number:
 1159:           type: number
 1160:           min: 0
 1161:       output:
 1162:         type: number
 1163:         min: 0
 1164: 
 1165:     negative:
 1166:       input:
 1167:         number:
 1168:           type: number
 1169:           max: 0
 1170:       output:
 1171:         type: number
 1172:         min: 0

 1174: This configuration:
 1175: 
 1176: =over 4
 1177: 
 1178: =item * Enables property-based testing with 1000 trials per property
 1179: 
 1180: =item * Defines two transforms: one for positive numbers, one for negative
 1181: 
 1182: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1183: 
 1184: =back
 1185: 
 1186: Generate the test:
 1187: 
 1188:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t

 1191: 
 1192: =over 4
 1193: 
 1194: =item * Traditional edge-case tests for boundary conditions
 1195: 
 1196: =item * Random fuzzing with 30 iterations (or as configured)
 1197: 
 1198: =item * Property-based tests that verify the transforms with 1000 trials each
 1199: 
 1200: =back

 1202: =head3 What Properties Are Tested?
 1203: 
 1204: The generator automatically detects and tests these properties based on your transform specifications:
 1205: 
 1206: =over 4
 1207: 
 1208: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1209: 
 1210: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1211: 
 1212: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1213: 
 1214: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
 1215: 
 1216: =back

 1219: 
 1220:   # For the "positive" transform:
 1221:   - Given a positive number, abs() returns >= 0
 1222:   - The result is a valid number
 1223:   - The result is defined
 1224: 
 1225:   # For the "negative" transform:
 1226:   - Given a negative number, abs() returns >= 0
 1227:   - The result is a valid number
 1228:   - The result is defined
