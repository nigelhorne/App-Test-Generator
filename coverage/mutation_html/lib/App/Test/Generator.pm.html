  155: =head1 OVERVIEW
  156: 
  157: This module takes a formal input/output specification for a routine or
  158: method and automatically generates test cases. In effect, it allows you
  159: to easily add comprehensive black-box tests in addition to the more
  160: common white-box tests that are typically written for CPAN modules and other
  161: subroutines.
  162: 
  163: The generated tests combine:
  164: 
  165: =over 4
  166: 
  167: =item * Random fuzzing based on input types
  168: 
  169: =item * Deterministic edge cases for min/max constraints
  170: 
  171: =item * Static corpus tests defined in Perl or YAML
  172: 
  173: =back
  174: 
  175: This approach strengthens your test suite by probing both expected and
  176: unexpected inputs, helping you to catch boundary errors, invalid data
  177: handling, and regressions without manually writing every case.
  178: 
  179: =head1 DESCRIPTION
  180: 
  181: This module implements the logic behind L<fuzz-harness-generator>.
  182: It parses configuration files (fuzz and/or corpus YAML), and
  183: produces a ready-to-run F<.t> test script to run through C<prove>.
  184: 
  185: It reads configuration files in any format,
  186: and optional YAML corpus files.
  187: All of the examples in this documentation are in C<YAML> format,
  188: other formats may not work as they aren't so heavily tested.
  189: It then generates a L<Test::Most>-based fuzzing harness combining:
  190: 
  191: =over 4
  192: 
  193: =item * Randomized fuzzing of inputs (with edge cases)
  194: 
  195: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  196: 
  197: =item * Functional or OO mode (via C<$new>)
  198: 
  199: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  200: 
  201: =back
  202: 
  203: =head1 MUTATION-GUIDED TEST GENERATION
  204: 
  205: C<App::Test::Generator> includes a pipeline that automatically closes the
  206: feedback loop between mutation testing, schema extraction, and fuzz
  207: testing. The goal is that surviving mutants drive the creation of new
  208: tests that kill them on the next run, without manual intervention.
  209: 
  210: =head2 The Pipeline
  211: 
  212:     mutation survivor
  213:         |
  214:         v
  215:     SchemaExtractor extracts the schema for the enclosing sub
  216:         |
  217:         v
  218:     Schema augmented with boundary values from the mutant
  219:         |
  220:         v
  221:     Augmented schema written to t/conf/
  222:         |
  223:         v
  224:     t/fuzz.t picks up the new schema and runs fuzz tests
  225:         |
  226:         v
  227:     Mutation killed on next run
  228: 
  229: =head2 How to Use It
  230: 
  231: The pipeline is driven by three flags passed to
  232: C<bin/app-test-generator-index>, which is invoked automatically by
  233: C<bin/generate-test-dashboard> on each CI push.
  234: 
  235: =head3 Step 1: Generate TODO stubs for all survivors
  236: 
  237:     bin/app-test-generator-index --generate_mutant_tests=t
  238: 
  239: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  240: 
  241: =over 4
  242: 
  243: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  244: boundary value suggestions, environment variable hints, and the
  245: enclosing subroutine name for navigation context.
  246: 
  247: =item * Comment-only hints for LOW difficulty survivors.
  248: 
  249: =back
  250: 
  251: Multiple mutations on the same source line are deduplicated into one
  252: stub. One good test kills all variants on that line.
  253: 
  254: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  255: 
  256:     bin/app-test-generator-index \
  257:         --generate_mutant_tests=t \
  258:         --generate_test=mutant
  259: 
  260: For each NUM_BOUNDARY survivor, calls
  261: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  262: the enclosing subroutine. If the confidence level is sufficient, the
  263: schema is augmented with the boundary value from the mutant (plus one
  264: value either side) and written to C<t/conf/> as a runnable YAML file.
  265: L<t/fuzz.t> picks it up automatically on the next test run.
  266: 
  267: Falls back to a TODO stub if:
  268: 
  269: =over 4
  270: 
  271: =item * SchemaExtractor cannot parse the file
  272: 
  273: =item * The enclosing sub cannot be determined
  274: 
  275: =item * The extracted schema confidence is C<very_low> or C<none>
  276: 
  277: =back
  278: 
  279: =head3 Step 3: Augment existing schemas with survivor boundary values
  280: 
  281:     bin/app-test-generator-index \
  282:         --generate_mutant_tests=t \
  283:         --generate_test=mutant \
  284:         --generate_fuzz
  285: 
  286: Scans C<t/conf/> for existing YAML schema files (hand-written or
  287: previously generated) and writes augmented copies with boundary values
  288: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  289: never modified. Augmented copies are written as
  290: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  291: automatically by C<t/fuzz.t>.
  292: 
  293: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  294: to prevent cascading augmentation. Schemas with no matching survivors
  295: are skipped, with a note if C<--verbose> is active.
  296: 
  297: =head3 Putting It All Together
  298: 
  299: The recommended invocation in C<bin/generate-test-dashboard>
  300: Step 7 runs all three stages together:
  301: 
  302:     bin/app-test-generator-index \
  303:         --generate_mutant_tests=t \
  304:         --generate_test=mutant \
  305:         --generate_fuzz
  306: 
  307: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  308: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  309: to the repository so they accumulate over time as the test suite
  310: improves.

  315: to each extracted schema:
  316: 
  317: =over 4
  318: 
  319: =item * C<high> / C<medium> / C<low> - Schema is used for test generation

  375: 
  376: The schema is split into several sections.
  377: 
  378: =head3 C<%input> - input params with keys => type/optional specs
  379: 
  380: When using named parameters
  381: 
  382:   input:
  383:     name:
  384:       type: string
  385:       optional: false
  386:     age:
  387:       type: integer
  388:       optional: true
  389: 
  390: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  391: See also L<Params::Validate::Strict>.
  392: You can add more custom types using properties.
  393: 
  394: For routines with one unnamed parameter
  395: 
  396:   input:
  397:     type: string
  398: 
  399: For routines with more than one named parameter, use the C<position> keyword.
  400: 
  401:   module: Math::Simple::MinMax
  402:   fuction: max
  403: 
  404:   input:
  405:     left:
  406:       type: number
  407:       position: 0
  408:     right:
  409:       type: number
  410:       position: 1
  411: 
  412:   output:
  413:     type: number
  414: 
  415: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  416: 
  417: =head3 C<%output> - output param types for L<Return::Set> checking
  418: 
  419:   output:
  420:     type: string
  421: 
  422: If the output hash contains the key _STATUS, and if that key is set to DIES,
  423: the routine should die with the given arguments; otherwise, it should live.
  424: If it's set to WARNS,
  425: the routine should warn with the given arguments.
  426: The output can be set to the string 'undef' if the routine should return the undefined value:
  427: 
  428:   ---
  429:   module: Scalar::Util
  430:   function: blessed
  431: 
  432:   input:
  433:     type: string
  434: 
  435:   output: undef
  436: 
  437: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  438: 
  439: =head3 C<%config> - optional hash of configuration.
  440: 
  441: The current supported variables are
  442: 
  443: =over 4
  444: 
  445: =item * C<close_stdin>
  446: 
  447: Tests should not attempt to read from STDIN (default: 1).
  448: This is ignored on Windows, when never closes STDIN.
  449: 
  450: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  451: 
  452: With this test enabled, the function is expected to die when a NUL byte is passed in.
  453: 
  454: =item * C<test_undef>, test with undefined value (default: 1)
  455: 
  456: =item * C<test_empty>, test with empty strings (default: 1)
  457: 
  458: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  459: 
  460: =item * C<timeout>, ensure tests don't hang (default: 10)
  461: 
  462: Setting this to 0 disables timeout testing.
  463: 
  464: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  465: 
  466: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  467: 
  468: *item * C<test_security>, send some security string based tests (default: 0)
  469: 
  470: =back
  471: 
  472: All values default to C<true>.

  475: 
  476:   accessor:
  477:     property: ua
  478:     type: getset
  479: 
  480: Has two mandatory elements:
  481: 
  482: =over 4
  483: 
  484: =item * C<property>
  485: 
  486: The name of the property in the object that the routine controls.
  487: 
  488: =item * C<type>

  494: =head3 C<%transforms> - list of transformations from input sets to output sets
  495: 
  496: Transforms allow you to define how input data should be transformed into output data.
  497: This is useful for testing functions that convert between formats, normalize data,
  498: or apply business logic transformations on a set of data to different set of data.
  499: It takes a list of subsets of the input and output definitions,
  500: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  501: 
  502: =head4 Transform Validation Rules
  503: 
  504: For each transform:
  505: 
  506: =over 4
  507: 
  508: =item 1. Generate test cases using the transform's input schema
  509: 
  510: =item 2. Call the function with those inputs
  511: 
  512: =item 3. Validate the output matches the transform's output schema
  513: 
  514: =item 4. If output has a specific 'value', check exact match
  515: 
  516: =item 5. If output has constraints (min/max), validate within bounds
  517: 
  518: =back

  787: =head2 EDGE CASE GENERATION
  788: 
  789: In addition to purely random fuzz cases, the harness generates
  790: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  791: 
  792: For each constraint, three edge cases are added:
  793: 
  794: =over 4
  795: 
  796: =item * Just inside the allowable range
  797: 
  798: This case should succeed, since it lies strictly within the bounds.
  799: 
  800: =item * Exactly on the boundary
  801: 
  802: This case should succeed, since it meets the constraint exactly.
  803: 
  804: =item * Just outside the boundary
  805: 
  806: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  807: should cause the harness to fail validation or croak.
  808: 
  809: =back
  810: 
  811: Supported constraint types:
  812: 
  813: =over 4
  814: 
  815: =item * C<number>, C<integer>, C<float>
  816: 
  817: Uses numeric values one below, equal to, and one above the boundary.
  818: 
  819: =item * C<string>
  820: 
  821: Uses strings of lengths one below, equal to, and one above the boundary.
  822: 
  823: =item * C<arrayref>
  824: 
  825: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  826: 
  827: =item * C<hashref>
  828: 
  829: Uses hashes with key counts one below, equal to, and one above the
  830: boundary (C<min> = minimum number of keys, C<max> = maximum number
  831: of keys).
  832: 
  833: =item * C<memberof> - arrayref of allowed values for a parameter
  834: 
  835: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  836: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  837: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  838: 
  839:   ---
  840:   input:
  841:     status:
  842:       type: string
  843:       memberof:
  844:         - ok
  845:         - error
  846:         - pending
  847:     level:
  848:       type: integer
  849:       memberof:
  850:         - 1
  851:         - 5
  852:         - 111
  853: 
  854: The generator will automatically create test cases for each allowed value (inside the member list),
  855: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  856: This works for strings, integers, and numbers.
  857: 
  858: =item * C<enum> - synonym of C<memberof>
  859: 
  860: =item * C<boolean> - automatic boundary tests for boolean fields
  861: 
  862:   input:
  863:     flag:
  864:       type: boolean
  865: 
  866: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  867: 
  868: =back
  869: 
  870: These edge cases are inserted automatically, in addition to the random
  871: fuzzing inputs, so each run will reliably probe boundary conditions
  872: without relying solely on randomness.

 1032: =head2 Property-Based Testing with Transforms
 1033: 
 1034: The generator can create property-based tests using L<Test::LectroTest> when the
 1035: C<properties> configuration option is enabled.
 1036: This provides more comprehensive
 1037: testing by automatically generating thousands of test cases and verifying that
 1038: mathematical properties hold across all inputs.
 1039: 
 1040: =head3 Basic Property-Based Transform Example
 1041: 
 1042: Here's a complete example testing the C<abs> builtin function:
 1043: 
 1044: B<t/conf/abs.yml>:
 1045: 
 1046:   ---
 1047:   module: builtin
 1048:   function: abs
 1049: 
 1050:   config:
 1051:     test_undef: no
 1052:     test_empty: no
 1053:     test_nuls: no
 1054:     properties:
 1055:       enable: true
 1056:       trials: 1000
 1057: 
 1058:   input:
 1059:     number:
 1060:       type: number
 1061:       position: 0
 1062: 
 1063:   output:
 1064:     type: number
 1065:     min: 0
 1066: 
 1067:   transforms:
 1068:     positive:
 1069:       input:
 1070:         number:
 1071:           type: number
 1072:           min: 0
 1073:       output:
 1074:         type: number
 1075:         min: 0
 1076: 
 1077:     negative:
 1078:       input:
 1079:         number:
 1080:           type: number
 1081:           max: 0
 1082:       output:
 1083:         type: number
 1084:         min: 0
 1085: 
 1086: This configuration:
 1087: 
 1088: =over 4
 1089: 
 1090: =item * Enables property-based testing with 1000 trials per property
 1091: 
 1092: =item * Defines two transforms: one for positive numbers, one for negative
 1093: 
 1094: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1095: 
 1096: =back
 1097: 
 1098: Generate the test:
 1099: 
 1100:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t
 1101: 
 1102: The generated test will include:
 1103: 
 1104: =over 4
 1105: 
 1106: =item * Traditional edge-case tests for boundary conditions
 1107: 
 1108: =item * Random fuzzing with 30 iterations (or as configured)
 1109: 
 1110: =item * Property-based tests that verify the transforms with 1000 trials each
 1111: 
 1112: =back

 1114: =head3 What Properties Are Tested?
 1115: 
 1116: The generator automatically detects and tests these properties based on your transform specifications:
 1117: 
 1118: =over 4
 1119: 
 1120: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1121: 
 1122: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1123: 
 1124: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1125: 
 1126: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
 1127: 
 1128: =back
 1129: 
 1130: For the C<abs> example above, the generated properties verify:
 1131: 
 1132:   # For the "positive" transform:
 1133:   - Given a positive number, abs() returns >= 0
 1134:   - The result is a valid number
 1135:   - The result is defined
 1136: 
 1137:   # For the "negative" transform:
 1138:   - Given a negative number, abs() returns >= 0
 1139:   - The result is a valid number
 1140:   - The result is defined
