  161: =head1 OVERVIEW
  162: 
  163: This module takes a formal input/output specification for a routine or
  164: method and automatically generates test cases. In effect, it allows you
  165: to easily add comprehensive black-box tests in addition to the more
  166: common white-box tests that are typically written for CPAN modules and other
  167: subroutines.
  168: 
  169: The generated tests combine:
  170: 
  171: =over 4
  172: 
  173: =item * Random fuzzing based on input types
  174: 
  175: =item * Deterministic edge cases for min/max constraints
  176: 
  177: =item * Static corpus tests defined in Perl or YAML
  178: 
  179: =back
  180: 
  181: This approach strengthens your test suite by probing both expected and
  182: unexpected inputs, helping you to catch boundary errors, invalid data
  183: handling, and regressions without manually writing every case.
  184: 
  185: =head1 DESCRIPTION
  186: 
  187: This module implements the logic behind L<fuzz-harness-generator>.
  188: It parses configuration files (fuzz and/or corpus YAML), and
  189: produces a ready-to-run F<.t> test script to run through C<prove>.
  190: 
  191: It reads configuration files in any format,
  192: and optional YAML corpus files.
  193: All of the examples in this documentation are in C<YAML> format,
  194: other formats may not work as they aren't so heavily tested.
  195: It then generates a L<Test::Most>-based fuzzing harness combining:
  196: 
  197: =over 4
  198: 
  199: =item * Randomized fuzzing of inputs (with edge cases)
  200: 
  201: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  202: 
  203: =item * Functional or OO mode (via C<$new>)
  204: 
  205: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  206: 
  207: =back
  208: 
  209: =head1 MUTATION-GUIDED TEST GENERATION
  210: 
  211: C<App::Test::Generator> includes a pipeline that automatically closes the
  212: feedback loop between mutation testing, schema extraction, and fuzz
  213: testing. The goal is that surviving mutants drive the creation of new
  214: tests that kill them on the next run, without manual intervention.
  215: 
  216: =head2 The Pipeline
  217: 
  218:     mutation survivor
  219:         |
  220:         v
  221:     SchemaExtractor extracts the schema for the enclosing sub
  222:         |
  223:         v
  224:     Schema augmented with boundary values from the mutant
  225:         |
  226:         v
  227:     Augmented schema written to t/conf/
  228:         |
  229:         v
  230:     t/fuzz.t picks up the new schema and runs fuzz tests
  231:         |
  232:         v
  233:     Mutation killed on next run
  234: 
  235: =head2 How to Use It
  236: 
  237: The pipeline is driven by three flags passed to
  238: C<bin/app-test-generator-index>, which is invoked automatically by
  239: C<bin/generate-test-dashboard> on each CI push.
  240: 
  241: =head3 Step 1: Generate TODO stubs for all survivors
  242: 
  243:     bin/app-test-generator-index --generate_mutant_tests=t
  244: 
  245: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  246: 
  247: =over 4
  248: 
  249: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  250: boundary value suggestions, environment variable hints, and the
  251: enclosing subroutine name for navigation context.
  252: 
  253: =item * Comment-only hints for LOW difficulty survivors.
  254: 
  255: =back
  256: 
  257: Multiple mutations on the same source line are deduplicated into one
  258: stub. One good test kills all variants on that line.
  259: 
  260: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  261: 
  262:     bin/app-test-generator-index \
  263:         --generate_mutant_tests=t \
  264:         --generate_test=mutant
  265: 
  266: For each NUM_BOUNDARY survivor, calls
  267: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  268: the enclosing subroutine. If the confidence level is sufficient, the
  269: schema is augmented with the boundary value from the mutant (plus one
  270: value either side) and written to C<t/conf/> as a runnable YAML file.
  271: L<t/fuzz.t> picks it up automatically on the next test run.
  272: 
  273: Falls back to a TODO stub if:
  274: 
  275: =over 4
  276: 
  277: =item * SchemaExtractor cannot parse the file
  278: 
  279: =item * The enclosing sub cannot be determined
  280: 
  281: =item * The extracted schema confidence is C<very_low> or C<none>
  282: 
  283: =back
  284: 
  285: =head3 Step 3: Augment existing schemas with survivor boundary values
  286: 
  287:     bin/app-test-generator-index \
  288:         --generate_mutant_tests=t \
  289:         --generate_test=mutant \
  290:         --generate_fuzz
  291: 
  292: Scans C<t/conf/> for existing YAML schema files (hand-written or
  293: previously generated) and writes augmented copies with boundary values
  294: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  295: never modified. Augmented copies are written as
  296: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  297: automatically by C<t/fuzz.t>.
  298: 
  299: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  300: to prevent cascading augmentation. Schemas with no matching survivors
  301: are skipped, with a note if C<--verbose> is active.
  302: 
  303: =head3 Putting It All Together
  304: 
  305: The recommended invocation in C<bin/generate-test-dashboard>
  306: Step 7 runs all three stages together:
  307: 
  308:     bin/app-test-generator-index \
  309:         --generate_mutant_tests=t \
  310:         --generate_test=mutant \
  311:         --generate_fuzz
  312: 
  313: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  314: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  315: to the repository so they accumulate over time as the test suite
  316: improves.

  319: 
  320: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  321: to each extracted schema:
  322: 
  323: =over 4
  324: 
  325: =item * C<high> / C<medium> / C<low> - Schema is used for test generation
  326: 
  327: =item * C<very_low> / C<none> - Falls back to TODO stub
  328: 
  329: =back

  375: 
  376: The configuration file,
  377: for each set of tests to be produced,
  378: is a file containing a schema that can be read by L<Config::Abstraction>.

  380: =head2 SCHEMA
  381: 
  382: The schema is split into several sections.
  383: 
  384: =head3 C<%input> - input params with keys => type/optional specs
  385: 
  386: When using named parameters
  387: 
  388:   input:
  389:     name:
  390:       type: string
  391:       optional: false
  392:     age:
  393:       type: integer
  394:       optional: true
  395: 
  396: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  397: See also L<Params::Validate::Strict>.
  398: You can add more custom types using properties.
  399: 
  400: For routines with one unnamed parameter
  401: 
  402:   input:
  403:     type: string
  404: 
  405: For routines with more than one named parameter, use the C<position> keyword.
  406: 
  407:   module: Math::Simple::MinMax
  408:   fuction: max
  409: 
  410:   input:
  411:     left:
  412:       type: number
  413:       position: 0
  414:     right:
  415:       type: number
  416:       position: 1
  417: 
  418:   output:
  419:     type: number
  420: 
  421: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  422: 
  423: =head3 C<%output> - output param types for L<Return::Set> checking
  424: 
  425:   output:
  426:     type: string
  427: 
  428: If the output hash contains the key _STATUS, and if that key is set to DIES,
  429: the routine should die with the given arguments; otherwise, it should live.
  430: If it's set to WARNS,
  431: the routine should warn with the given arguments.
  432: The output can be set to the string 'undef' if the routine should return the undefined value:
  433: 
  434:   ---
  435:   module: Scalar::Util
  436:   function: blessed
  437: 
  438:   input:
  439:     type: string
  440: 
  441:   output: undef
  442: 
  443: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  444: 
  445: =head3 C<%config> - optional hash of configuration.
  446: 
  447: The current supported variables are
  448: 
  449: =over 4
  450: 
  451: =item * C<close_stdin>
  452: 
  453: Tests should not attempt to read from STDIN (default: 1).
  454: This is ignored on Windows, when never closes STDIN.
  455: 
  456: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  457: 
  458: With this test enabled, the function is expected to die when a NUL byte is passed in.
  459: 
  460: =item * C<test_undef>, test with undefined value (default: 1)
  461: 
  462: =item * C<test_empty>, test with empty strings (default: 1)
  463: 
  464: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  465: 
  466: =item * C<timeout>, ensure tests don't hang (default: 10)
  467: 
  468: Setting this to 0 disables timeout testing.
  469: 
  470: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  471: 
  472: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  473: 
  474: *item * C<test_security>, send some security string based tests (default: 0)
  475: 
  476: =back
  477: 
  478: All values default to C<true>.
  479: 
  480: =head3 C<%accessor> - this is an accessor routine
  481: 
  482:   accessor:
  483:     property: ua
  484:     type: getset
  485: 
  486: Has two mandatory elements:
  487: 
  488: =over 4
  489: 
  490: =item * C<property>
  491: 
  492: The name of the property in the object that the routine controls.
  493: 
  494: =item * C<type>
  495: 
  496: One of C<getter>, C<setter>, C<getset>.
  497: 
  498: =back
  499: 
  500: =head3 C<%transforms> - list of transformations from input sets to output sets
  501: 
  502: Transforms allow you to define how input data should be transformed into output data.
  503: This is useful for testing functions that convert between formats, normalize data,
  504: or apply business logic transformations on a set of data to different set of data.
  505: It takes a list of subsets of the input and output definitions,
  506: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  507: 
  508: =head4 Transform Validation Rules
  509: 
  510: For each transform:
  511: 
  512: =over 4
  513: 
  514: =item 1. Generate test cases using the transform's input schema
  515: 
  516: =item 2. Call the function with those inputs
  517: 
  518: =item 3. Validate the output matches the transform's output schema
  519: 
  520: =item 4. If output has a specific 'value', check exact match
  521: 
  522: =item 5. If output has constraints (min/max), validate within bounds
  523: 
  524: =back

  793: =head2 EDGE CASE GENERATION
  794: 
  795: In addition to purely random fuzz cases, the harness generates
  796: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  797: 
  798: For each constraint, three edge cases are added:
  799: 
  800: =over 4
  801: 
  802: =item * Just inside the allowable range
  803: 
  804: This case should succeed, since it lies strictly within the bounds.
  805: 
  806: =item * Exactly on the boundary
  807: 
  808: This case should succeed, since it meets the constraint exactly.
  809: 
  810: =item * Just outside the boundary
  811: 
  812: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  813: should cause the harness to fail validation or croak.
  814: 
  815: =back
  816: 
  817: Supported constraint types:
  818: 
  819: =over 4
  820: 
  821: =item * C<number>, C<integer>, C<float>
  822: 
  823: Uses numeric values one below, equal to, and one above the boundary.
  824: 
  825: =item * C<string>
  826: 
  827: Uses strings of lengths one below, equal to, and one above the boundary.
  828: 
  829: =item * C<arrayref>
  830: 
  831: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  832: 
  833: =item * C<hashref>
  834: 
  835: Uses hashes with key counts one below, equal to, and one above the
  836: boundary (C<min> = minimum number of keys, C<max> = maximum number
  837: of keys).
  838: 
  839: =item * C<memberof> - arrayref of allowed values for a parameter
  840: 
  841: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  842: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  843: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  844: 
  845:   ---
  846:   input:
  847:     status:
  848:       type: string
  849:       memberof:
  850:         - ok
  851:         - error
  852:         - pending
  853:     level:
  854:       type: integer
  855:       memberof:
  856:         - 1
  857:         - 5
  858:         - 111
  859: 
  860: The generator will automatically create test cases for each allowed value (inside the member list),
  861: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  862: This works for strings, integers, and numbers.
  863: 
  864: =item * C<enum> - synonym of C<memberof>
  865: 
  866: =item * C<boolean> - automatic boundary tests for boolean fields
  867: 
  868:   input:
  869:     flag:
  870:       type: boolean
  871: 
  872: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  873: 
  874: =back
  875: 
  876: These edge cases are inserted automatically, in addition to the random
  877: fuzzing inputs, so each run will reliably probe boundary conditions
  878: without relying solely on randomness.

 1038: =head2 Property-Based Testing with Transforms
 1039: 
 1040: The generator can create property-based tests using L<Test::LectroTest> when the
 1041: C<properties> configuration option is enabled.
 1042: This provides more comprehensive
 1043: testing by automatically generating thousands of test cases and verifying that
 1044: mathematical properties hold across all inputs.
 1045: 
 1046: =head3 Basic Property-Based Transform Example
 1047: 
 1048: Here's a complete example testing the C<abs> builtin function:
 1049: 
 1050: B<t/conf/abs.yml>:
 1051: 
 1052:   ---
 1053:   module: builtin
 1054:   function: abs
 1055: 
 1056:   config:
 1057:     test_undef: no
 1058:     test_empty: no
 1059:     test_nuls: no
 1060:     properties:
 1061:       enable: true
 1062:       trials: 1000
 1063: 
 1064:   input:
 1065:     number:
 1066:       type: number
 1067:       position: 0
 1068: 
 1069:   output:
 1070:     type: number
 1071:     min: 0
 1072: 
 1073:   transforms:
 1074:     positive:
 1075:       input:
 1076:         number:
 1077:           type: number
 1078:           min: 0
 1079:       output:
 1080:         type: number
 1081:         min: 0
 1082: 
 1083:     negative:
 1084:       input:
 1085:         number:
 1086:           type: number
 1087:           max: 0
 1088:       output:
 1089:         type: number
 1090:         min: 0
 1091: 
 1092: This configuration:
 1093: 
 1094: =over 4
 1095: 
 1096: =item * Enables property-based testing with 1000 trials per property
 1097: 
 1098: =item * Defines two transforms: one for positive numbers, one for negative
 1099: 
 1100: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1101: 
 1102: =back
 1103: 
 1104: Generate the test:
 1105: 
 1106:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t
 1107: 
 1108: The generated test will include:
 1109: 
 1110: =over 4
 1111: 
 1112: =item * Traditional edge-case tests for boundary conditions
 1113: 
 1114: =item * Random fuzzing with 30 iterations (or as configured)
 1115: 
 1116: =item * Property-based tests that verify the transforms with 1000 trials each
 1117: 
 1118: =back
 1119: 
 1120: =head3 What Properties Are Tested?
 1121: 
 1122: The generator automatically detects and tests these properties based on your transform specifications:
 1123: 
 1124: =over 4
 1125: 
 1126: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1127: 
 1128: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1129: 
 1130: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1131: 
 1132: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
 1133: 
 1134: =back
 1135: 
 1136: For the C<abs> example above, the generated properties verify:
 1137: 
 1138:   # For the "positive" transform:
 1139:   - Given a positive number, abs() returns >= 0
 1140:   - The result is a valid number
 1141:   - The result is defined
 1142: 
 1143:   # For the "negative" transform:
 1144:   - Given a negative number, abs() returns >= 0
 1145:   - The result is a valid number
 1146:   - The result is defined
