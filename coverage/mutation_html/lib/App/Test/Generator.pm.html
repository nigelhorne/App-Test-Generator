  168: =head1 OVERVIEW
  169: 
  170: This module takes a formal input/output specification for a routine or
  171: method and automatically generates test cases. In effect, it allows you
  172: to easily add comprehensive black-box tests in addition to the more
  173: common white-box tests that are typically written for CPAN modules and other
  174: subroutines.
  175: 
  176: The generated tests combine:
  177: 
  178: =over 4
  179: 
  180: =item * Random fuzzing based on input types
  181: 
  182: =item * Deterministic edge cases for min/max constraints
  183: 
  184: =item * Static corpus tests defined in Perl or YAML
  185: 
  186: =back
  187: 
  188: This approach strengthens your test suite by probing both expected and
  189: unexpected inputs, helping you to catch boundary errors, invalid data
  190: handling, and regressions without manually writing every case.
  191: 
  192: =head1 DESCRIPTION
  193: 
  194: This module implements the logic behind L<fuzz-harness-generator>.
  195: It parses configuration files (fuzz and/or corpus YAML), and
  196: produces a ready-to-run F<.t> test script to run through C<prove>.
  197: 
  198: It reads configuration files in any format,
  199: and optional YAML corpus files.
  200: All of the examples in this documentation are in C<YAML> format,
  201: other formats may not work as they aren't so heavily tested.
  202: It then generates a L<Test::Most>-based fuzzing harness combining:
  203: 
  204: =over 4
  205: 
  206: =item * Randomized fuzzing of inputs (with edge cases)
  207: 
  208: =item * Optional static corpus tests from Perl C<%cases> or YAML file (C<yaml_cases> key)
  209: 
  210: =item * Functional or OO mode (via C<$new>)
  211: 
  212: =item * Reproducible runs via C<$seed> and configurable iterations via C<$iterations>
  213: 
  214: =back
  215: 
  216: =head1 MUTATION-GUIDED TEST GENERATION
  217: 
  218: C<App::Test::Generator> includes a pipeline that automatically closes the
  219: feedback loop between mutation testing, schema extraction, and fuzz
  220: testing. The goal is that surviving mutants drive the creation of new
  221: tests that kill them on the next run, without manual intervention.
  222: 
  223: =head2 The Pipeline
  224: 
  225:     mutation survivor
  226:         |
  227:         v
  228:     SchemaExtractor extracts the schema for the enclosing sub
  229:         |
  230:         v
  231:     Schema augmented with boundary values from the mutant
  232:         |
  233:         v
  234:     Augmented schema written to t/conf/
  235:         |
  236:         v
  237:     t/fuzz.t picks up the new schema and runs fuzz tests
  238:         |
  239:         v
  240:     Mutation killed on next run
  241: 
  242: =head2 How to Use It
  243: 
  244: The pipeline is driven by three flags passed to
  245: C<bin/app-test-generator-index>, which is invoked automatically by
  246: C<bin/generate-test-dashboard> on each CI push.
  247: 
  248: =head3 Step 1: Generate TODO stubs for all survivors
  249: 
  250:     bin/app-test-generator-index --generate_mutant_tests=t
  251: 
  252: Produces C<t/mutant_YYYYMMDD_HHMMSS.t> containing:
  253: 
  254: =over 4
  255: 
  256: =item * TODO stubs for HIGH and MEDIUM difficulty survivors, with
  257: boundary value suggestions, environment variable hints, and the
  258: enclosing subroutine name for navigation context.
  259: 
  260: =item * Comment-only hints for LOW difficulty survivors.
  261: 
  262: =back
  263: 
  264: Multiple mutations on the same source line are deduplicated into one
  265: stub. One good test kills all variants on that line.
  266: 
  267: =head3 Step 2: Generate runnable schemas for NUM_BOUNDARY survivors
  268: 
  269:     bin/app-test-generator-index \
  270:         --generate_mutant_tests=t \
  271:         --generate_test=mutant
  272: 
  273: For each NUM_BOUNDARY survivor, calls
  274: L<App::Test::Generator::SchemaExtractor> to extract the schema for
  275: the enclosing subroutine. If the confidence level is sufficient, the
  276: schema is augmented with the boundary value from the mutant (plus one
  277: value either side) and written to C<t/conf/> as a runnable YAML file.
  278: L<t/fuzz.t> picks it up automatically on the next test run.
  279: 
  280: Falls back to a TODO stub if:
  281: 
  282: =over 4
  283: 
  284: =item * SchemaExtractor cannot parse the file
  285: 
  286: =item * The enclosing sub cannot be determined
  287: 
  288: =item * The extracted schema confidence is C<very_low> or C<none>
  289: 
  290: =back
  291: 
  292: =head3 Step 3: Augment existing schemas with survivor boundary values
  293: 
  294:     bin/app-test-generator-index \
  295:         --generate_mutant_tests=t \
  296:         --generate_test=mutant \
  297:         --generate_fuzz
  298: 
  299: Scans C<t/conf/> for existing YAML schema files (hand-written or
  300: previously generated) and writes augmented copies with boundary values
  301: from surviving NUM_BOUNDARY mutants merged in. The original schema is
  302: never modified. Augmented copies are written as
  303: C<t/conf/mutant_fuzz_YYYYMMDD_HHMMSS_FUNCTION.yml> and picked up
  304: automatically by C<t/fuzz.t>.
  305: 
  306: Schemas whose filename already starts with C<mutant_fuzz_> are skipped
  307: to prevent cascading augmentation. Schemas with no matching survivors
  308: are skipped, with a note if C<--verbose> is active.
  309: 
  310: =head3 Putting It All Together
  311: 
  312: The recommended invocation in C<bin/generate-test-dashboard>
  313: Step 7 runs all three stages together:
  314: 
  315:     bin/app-test-generator-index \
  316:         --generate_mutant_tests=t \
  317:         --generate_test=mutant \
  318:         --generate_fuzz
  319: 
  320: The GitHub Actions workflow in C<.github/workflows/dashboard.yml>
  321: then commits any new C<t/mutant_*.t> and C<t/conf/mutant_*.yml> files
  322: to the repository so they accumulate over time as the test suite
  323: improves.

  325: =head2 Confidence Levels
  326: 
  327: L<App::Test::Generator::SchemaExtractor> assigns a confidence level
  328: to each extracted schema:
  329: 
  330: =over 4
  331: 
  332: =item * C<high> / C<medium> / C<low> - Schema is used for test generation
  333: 
  334: =item * C<very_low> / C<none> - Falls back to TODO stub
  335: 
  336: =back
  337: 
  338: Confidence is based on how much type and constraint information could
  339: be inferred from the source code and its POD documentation. Methods
  340: with explicit parameter validation (L<Params::Validate::Strict>,
  341: L<Params::Get>) or comprehensive POD will produce higher-confidence
  342: schemas.

  382: 
  383: The configuration file,
  384: for each set of tests to be produced,
  385: is a file containing a schema that can be read by L<Config::Abstraction>.

  387: =head2 SCHEMA
  388: 
  389: The schema is split into several sections.
  390: 
  391: =head3 C<%input> - input params with keys => type/optional specs
  392: 
  393: When using named parameters
  394: 
  395:   input:
  396:     name:
  397:       type: string
  398:       optional: false
  399:     age:
  400:       type: integer
  401:       optional: true
  402: 
  403: Supported basic types used by the fuzzer: C<string>, C<integer>, C<float>, C<number>, C<boolean>, C<arrayref>, C<hashref>.
  404: See also L<Params::Validate::Strict>.
  405: You can add more custom types using properties.
  406: 
  407: For routines with one unnamed parameter
  408: 
  409:   input:
  410:     type: string
  411: 
  412: For routines with more than one named parameter, use the C<position> keyword.
  413: 
  414:   module: Math::Simple::MinMax
  415:   fuction: max
  416: 
  417:   input:
  418:     left:
  419:       type: number
  420:       position: 0
  421:     right:
  422:       type: number
  423:       position: 1
  424: 
  425:   output:
  426:     type: number
  427: 
  428: The keyword C<undef> is used to indicate that the C<function> takes no arguments.
  429: 
  430: =head3 C<%output> - output param types for L<Return::Set> checking
  431: 
  432:   output:
  433:     type: string
  434: 
  435: If the output hash contains the key _STATUS, and if that key is set to DIES,
  436: the routine should die with the given arguments; otherwise, it should live.
  437: If it's set to WARNS,
  438: the routine should warn with the given arguments.
  439: The output can be set to the string 'undef' if the routine should return the undefined value:
  440: 
  441:   ---
  442:   module: Scalar::Util
  443:   function: blessed
  444: 
  445:   input:
  446:     type: string
  447: 
  448:   output: undef
  449: 
  450: The keyword C<undef> is used to indicate that the C<function> returns nothing.
  451: 
  452: =head3 C<%config> - optional hash of configuration.
  453: 
  454: The current supported variables are
  455: 
  456: =over 4
  457: 
  458: =item * C<close_stdin>
  459: 
  460: Tests should not attempt to read from STDIN (default: 1).
  461: This is ignored on Windows, when never closes STDIN.
  462: 
  463: =item * C<test_nuls>, inject NUL bytes into strings (default: 1)
  464: 
  465: With this test enabled, the function is expected to die when a NUL byte is passed in.
  466: 
  467: =item * C<test_undef>, test with undefined value (default: 1)
  468: 
  469: =item * C<test_empty>, test with empty strings (default: 1)
  470: 
  471: =item * C<test_non_ascii>, test with strings that contain non ascii characters (default: 1)
  472: 
  473: =item * C<timeout>, ensure tests don't hang (default: 10)
  474: 
  475: Setting this to 0 disables timeout testing.
  476: 
  477: =item * C<dedup>, fuzzing can create duplicate tests, go some way to remove duplicates (default: 1)
  478: 
  479: =item * C<properties>, enable L<Test::LectroTest> Property tests (default: 0)
  480: 
  481: *item * C<test_security>, send some security string based tests (default: 0)
  482: 
  483: =back
  484: 
  485: All values default to C<true>.
  486: 
  487: =head3 C<%accessor> - this is an accessor routine
  488: 
  489:   accessor:
  490:     property: ua
  491:     type: getset
  492: 
  493: Has two mandatory elements:
  494: 
  495: =over 4
  496: 
  497: =item * C<property>
  498: 
  499: The name of the property in the object that the routine controls.
  500: 
  501: =item * C<type>
  502: 
  503: One of C<getter>, C<setter>, C<getset>.
  504: 
  505: =back
  506: 
  507: =head3 C<%transforms> - list of transformations from input sets to output sets
  508: 
  509: Transforms allow you to define how input data should be transformed into output data.
  510: This is useful for testing functions that convert between formats, normalize data,
  511: or apply business logic transformations on a set of data to different set of data.
  512: It takes a list of subsets of the input and output definitions,
  513: and verifies that data from each input subset is correctly transformed into data from the matching output subset.
  514: 
  515: =head4 Transform Validation Rules
  516: 
  517: For each transform:
  518: 
  519: =over 4
  520: 
  521: =item 1. Generate test cases using the transform's input schema
  522: 
  523: =item 2. Call the function with those inputs
  524: 
  525: =item 3. Validate the output matches the transform's output schema
  526: 
  527: =item 4. If output has a specific 'value', check exact match
  528: 
  529: =item 5. If output has constraints (min/max), validate within bounds
  530: 
  531: =back

  684: =head3 C<%edge_cases>
  685: 
  686: An optional hash mapping of extra values to inject.
  687: 
  688: 	# Two named parameters
  689: 	edge_cases:
  690: 		name: [ '', 'a' x 1024, \"\x{263A}" ]
  691: 		age: [ -1, 0, 99999999 ]
  692: 
  693: 	# Takes a string input
  694: 	edge_cases: [ 'foo', 'bar' ]
  695: 
  696: Values can be strings or numbers; strings will be properly quoted.
  697: Note that this only works with routines that take named parameters.
  698: 
  699: =head3 C<%type_edge_cases>
  700: 
  701: An optional hash mapping types to arrayrefs of extra values to try for any field of that type:
  702: 
  703: 	type_edge_cases:
  704: 		string: [ '', ' ', "\t", "\n", "\0", 'long' x 1024, chr(0x1F600) ]
  705: 		number: [ 0, 1.0, -1.0, 1e308, -1e308, 1e-308, -1e-308, 'NaN', 'Infinity' ]
  706: 		integer: [ 0, 1, -1, 2**31-1, -(2**31), 2**63-1, -(2**63) ]
  707: 
  708: =head3 C<%edge_case_array>
  709: 
  710: Specify edge case values for routines that accept a single unnamed parameter.
  711: This is specifically designed for simple functions that take one argument without a parameter name.
  712: These edge cases supplement the normal random string generation, ensuring specific problematic values are always tested.
  713: During fuzzing iterations, there's a 40% probability that a test case will use a value from edge_case_array instead of randomly generated data.
  714: 
  715:   ---

  722:     max: 1000
  723: 
  724:   edge_case_array:

  800: =head2 EDGE CASE GENERATION
  801: 
  802: In addition to purely random fuzz cases, the harness generates
  803: deterministic edge cases for parameters that declare C<min>, C<max> or C<len> in their schema definitions.
  804: 
  805: For each constraint, three edge cases are added:
  806: 
  807: =over 4
  808: 
  809: =item * Just inside the allowable range
  810: 
  811: This case should succeed, since it lies strictly within the bounds.
  812: 
  813: =item * Exactly on the boundary
  814: 
  815: This case should succeed, since it meets the constraint exactly.
  816: 
  817: =item * Just outside the boundary
  818: 
  819: This case is annotated with C<_STATUS = 'DIES'> in the corpus and
  820: should cause the harness to fail validation or croak.
  821: 
  822: =back
  823: 
  824: Supported constraint types:
  825: 
  826: =over 4
  827: 
  828: =item * C<number>, C<integer>, C<float>
  829: 
  830: Uses numeric values one below, equal to, and one above the boundary.
  831: 
  832: =item * C<string>
  833: 
  834: Uses strings of lengths one below, equal to, and one above the boundary.
  835: 
  836: =item * C<arrayref>
  837: 
  838: Uses references to arrays of with the number of elements one below, equal to, and one above the boundary.
  839: 
  840: =item * C<hashref>
  841: 
  842: Uses hashes with key counts one below, equal to, and one above the
  843: boundary (C<min> = minimum number of keys, C<max> = maximum number
  844: of keys).
  845: 
  846: =item * C<memberof> - arrayref of allowed values for a parameter
  847: 
  848: This example is for a routine called C<input()> that takes two arguments: C<status> and C<level>.
  849: C<status> is a string that must have the value C<ok>, C<error> or C<pending>.
  850: The C<level> argument is an integer that must be one of C<1>, C<5> or C<111>.
  851: 
  852:   ---
  853:   input:
  854:     status:
  855:       type: string
  856:       memberof:
  857:         - ok
  858:         - error
  859:         - pending
  860:     level:
  861:       type: integer
  862:       memberof:
  863:         - 1
  864:         - 5
  865:         - 111
  866: 
  867: The generator will automatically create test cases for each allowed value (inside the member list),
  868: and at least one value outside the list (which should die or C<croak>, C<_STATUS = 'DIES'>).
  869: This works for strings, integers, and numbers.
  870: 
  871: =item * C<enum> - synonym of C<memberof>
  872: 
  873: =item * C<boolean> - automatic boundary tests for boolean fields
  874: 
  875:   input:
  876:     flag:
  877:       type: boolean
  878: 
  879: The generator will automatically create test cases for 0 and 1; true and false; off and on, and values that should trigger C<_STATUS = 'DIES'>.
  880: 
  881: =back
  882: 
  883: These edge cases are inserted automatically, in addition to the random
  884: fuzzing inputs, so each run will reliably probe boundary conditions
  885: without relying solely on randomness.

 1045: =head2 Property-Based Testing with Transforms
 1046: 
 1047: The generator can create property-based tests using L<Test::LectroTest> when the
 1048: C<properties> configuration option is enabled.
 1049: This provides more comprehensive
 1050: testing by automatically generating thousands of test cases and verifying that
 1051: mathematical properties hold across all inputs.
 1052: 
 1053: =head3 Basic Property-Based Transform Example
 1054: 
 1055: Here's a complete example testing the C<abs> builtin function:
 1056: 
 1057: B<t/conf/abs.yml>:
 1058: 
 1059:   ---
 1060:   module: builtin
 1061:   function: abs
 1062: 
 1063:   config:
 1064:     test_undef: no
 1065:     test_empty: no
 1066:     test_nuls: no
 1067:     properties:
 1068:       enable: true
 1069:       trials: 1000
 1070: 
 1071:   input:
 1072:     number:
 1073:       type: number
 1074:       position: 0
 1075: 
 1076:   output:
 1077:     type: number
 1078:     min: 0
 1079: 
 1080:   transforms:
 1081:     positive:
 1082:       input:
 1083:         number:
 1084:           type: number
 1085:           min: 0
 1086:       output:
 1087:         type: number
 1088:         min: 0
 1089: 
 1090:     negative:
 1091:       input:
 1092:         number:
 1093:           type: number
 1094:           max: 0
 1095:       output:
 1096:         type: number
 1097:         min: 0
 1098: 
 1099: This configuration:
 1100: 
 1101: =over 4
 1102: 
 1103: =item * Enables property-based testing with 1000 trials per property
 1104: 
 1105: =item * Defines two transforms: one for positive numbers, one for negative
 1106: 
 1107: =item * Automatically generates properties that verify C<abs()> always returns non-negative numbers
 1108: 
 1109: =back
 1110: 
 1111: Generate the test:
 1112: 
 1113:   fuzz-harness-generator t/conf/abs.yml > t/abs_property.t
 1114: 
 1115: The generated test will include:
 1116: 
 1117: =over 4
 1118: 
 1119: =item * Traditional edge-case tests for boundary conditions
 1120: 
 1121: =item * Random fuzzing with 30 iterations (or as configured)
 1122: 
 1123: =item * Property-based tests that verify the transforms with 1000 trials each
 1124: 
 1125: =back
 1126: 
 1127: =head3 What Properties Are Tested?
 1128: 
 1129: The generator automatically detects and tests these properties based on your transform specifications:
 1130: 
 1131: =over 4
 1132: 
 1133: =item * B<Range constraints> - If output has C<min> or C<max>, verifies results stay within bounds
 1134: 
 1135: =item * B<Type preservation> - Ensures numeric inputs produce numeric outputs
 1136: 
 1137: =item * B<Definedness> - Verifies the function doesn't return C<undef> unexpectedly
 1138: 
 1139: =item * B<Specific values> - If output specifies a C<value>, checks exact equality
 1140: 
 1141: =back
 1142: 
 1143: For the C<abs> example above, the generated properties verify:
 1144: 
 1145:   # For the "positive" transform:
 1146:   - Given a positive number, abs() returns >= 0
 1147:   - The result is a valid number
 1148:   - The result is defined
 1149: 
 1150:   # For the "negative" transform:
 1151:   - Given a negative number, abs() returns >= 0
 1152:   - The result is a valid number
 1153:   - The result is defined

 1226: 
 1227: When property-based tests run, you'll see output like:
 1228: 
 1229:   ok 123 - negative property holds (1000 trials)

 1234: 
 1235:   not ok 123 - positive property holds (47 trials)

 1237:   # Reason: counterexample found
